# Week 10~11  Memory Hierarchy

> The memory hierarchy exists because the **memory wall** between main memory and the CPU constrains real processor performance. The higher a level is in the hierarchy, the more frequently it is accessed; each level serves as a cache for the one below it, with CPU registers at the very top. By exploiting **locality**, the machine creates the illusion that memory is simultaneously large, fast, and cheap.

</br>

# Random-Access Memory (RAM)

“Random” means that **the access time does not depend on the physical position of the target data inside RAM**. By contrast, magnetic tape and other sequential-access media require scanning from the beginning, which is much less efficient.

- **DRAM = Dynamic RAM**: roughly **one capacitor plus one transistor per bit**. The capacitor stores the bit as “charged / uncharged,” but it leaks, so the memory must be **periodically refreshed**. DRAM is arranged as a 2D array with rows (RAS) and columns (CAS), sometimes described as “two halves.” Its canonical use is main memory: <u>slow, cheap, and large</u>. Main memory sits outside the CPU and is only **temporary** storage; files live on disk until they are needed, at which point they are cached into DRAM.

- **SRAM = Static RAM**: roughly **6 or 8 transistors plus resistive / latch structure per bit**. It stores information through the bistable state of a **flip-flop circuit**. As long as power remains on, the value persists. Its canonical uses are CPU caches and registers: <u>fast, expensive, and small</u>.

- DRAM is constrained by capacitor technology, while SRAM is constrained by semiconductor scaling. Both are approaching serious limits, even as processors continue to improve.

> SRAM often pays something like 100× the cost for roughly 10× lower access latency.

</br>

## DRAM

A DRAM array behaves like a two-dimensional matrix. It first selects a row using **RAS (row access strobe)**, then selects a column using **CAS (column access strobe)**. The row is copied into the row buffer, and the desired data is then delivered from there.

A **d × w DRAM** organization means that the total `d·w` bits are arranged into `d` **supercells**, each of width `w` bits. Here `w` is the data width transferred in one read or write, often **1 byte**.

- The address-port width is determined by how many rows and columns must be selected; the data-port width is **w**.
- On access: (1) issue RAS to select a row and copy it into the **row buffer**; (2) issue CAS to select a column and move the chosen supercell onto the data lines; (3) write the row back and refresh as needed.

**Memory modules** are the common modern organization of DRAM. For example, a 64 MB module may consist of eight 8M×8 DRAM chips, which collectively deliver a full 64-bit word.

- **DDR SDRAM** dominates the modern market. Other DRAM families may offer similar latency but higher bandwidth.
- **Non-volatile memory** such as flash is not DRAM. It retains data when power is removed, and belongs to the EEPROM family.
- **SAM** is typical of long-term storage such as tape-like sequential media.

![](https://i.postimg.cc/XvRSzj4t/jie-ping2026-03-26-08-09-22.png)

`lw`: the CPU places address `A` on the memory bus; main memory reads that address, retrieves word `x`, places `x` on the bus, and the CPU reads it into a register.

`sw`: the CPU places address `A` on the bus; memory prepares for a write; the CPU then copies value `y` from a register onto the bus; memory stores `y` into address `A`.

</br>

## The Memory Wall

> CPU performance has improved faster than DRAM and SSD technology, so the gap between processor speed and memory latency keeps widening. During the classic Moore’s-Law era, roughly from the 1980s to **2003**, this gap grew by about 50% per year. Before 2003 the widening was mainly a latency gap; afterward it increasingly became a broader performance gap.

The memory wall has two main ingredients: **latency**, because memory is physically far away from the CPU; and **bandwidth**, because the memory system and bus cannot feed data fast enough.

Main solution strategies:

> Historically, two major approaches emerged: **latency hiding**, such as multithreading, and **latency reduction**, such as caches. The latter proved cheaper and more influential on later machines.

- **Explicit local memory**: thread-private storage used when registers are insufficient.

- **Implicit memory hierarchy**: small, fast upper levels hold frequently used data, while large lower levels hold the full data set, creating the illusion of memory that is “large, cheap, and also fast.”

![](https://i.postimg.cc/3wV1jWYG/jie-ping2026-03-26-08-49-11.png)

- Recall that the **V-extension** provides vector registers, which can be used both for **SIMD** style **data-level parallelism** and for matrix-heavy tasks such as image processing.

The relationship between registers and main memory is controlled by the compiler or programmer; the movement between cache and memory is controlled by hardware; the movement between memory and disk is managed by hardware and the OS; and movement from disk to tape-like storage is controlled at higher software / system levels.

</br>

## Cache Definitions

> The dollar sign joke is unavoidable…

- In the broad sense, a cache is a **function**, not just the dedicated L1/L2/L3 structures. **A cache is not equivalent to SRAM.**

- **Block / line**: the transfer unit of a cache is a **memory block** or **cache line**. A line usually contains multiple words. On a miss, the cache fetches the entire block, not just one word.

- **Hit / miss**: if the requested data is already in the cache, it is a **hit**; otherwise, it is a **miss**. Important related quantities include:

- **hit ratio** = hits / accesses
- **miss ratio** = 1 - hit ratio
- **miss penalty** = the extra time paid on a miss
- **hit time** = the access time of the cache on a hit

> `hit time << miss penalty`, which is why even a miss rate of only a few percent can drastically hurt performance.

Caches work because they exploit two kinds of locality:

1. **Temporal locality**: after accessing A, the machine is likely to access A again soon.
2. **Spatial locality**: after accessing A, its neighbors — such as A−1 or A+1 — are also likely to be accessed.

#### Four classic cache-design questions:

Q1. Where can a block be placed?

Q2. How is a block found?

Q3. Which block should be replaced on a miss?

Q4. What happens on a write?

</br> 

</br>

# Cache Design

## Where can a block be placed?

**Associativity**:

- A **set** is the collection of cache lines into which a given memory block is allowed to map. In the usual model, the set is chosen by **`(block number) mod (#sets)`**.
- A **way** is the number of candidate lines inside each set.

**Fully associative cache**: effectively **one set**, so any memory block may go into any line. This reduces conflict misses, but tag lookup becomes expensive and the hardware becomes slower and more costly.

**Direct-mapped cache**: **one way per set, many sets**. Each memory block maps to exactly one line. This is simple, fast, and cheap, but more prone to **conflict misses**.

</br> 

## How is a block found?

- Since the cache is smaller than main memory, a memory address is divided into high bits (**tag**) and lower bits (**index** / set selector). Different memory blocks may map to the same index, so the cache must store the **tag** to distinguish them.

- We also need a **valid bit**: `V = 1` means the line contains valid data, while `V = 0` means the entry is empty, being filled, or invalidated.

For example, an **8-block direct-mapped cache** with 1 word per block has **8 sets**, so it uses a **3-bit index**. Suppose address 42 maps to index `010` and tag `101`. Then access proceeds as follows:

1. **Find the line and check hit/miss**: index into set `010`; initially the valid bit is 0, so this is a **compulsory miss**.
2. **Check the tag**: the desired tag `101` is absent, so it is still a miss.
3. **On a miss, fetch from main memory into that indexed line**, set the tag, and mark `V = 1`.
4. The next access to 42 becomes a hit.
5. If we later access 58 (`111010`), it maps to the same index and causes a **conflict miss**, replacing the earlier block.

- A **compulsory miss** is the first-ever access; a **conflict miss** occurs when two blocks compete for the same cache location.



### ⚠️ The compromise: Set-Associative Caches

> In an N-way cache, each set contains N candidate cache lines for the mapped block.

A block first maps to a set, and then may occupy any of the ways inside that set. Compared with the two extremes — direct-mapped and fully associative — N-way set-associative caches reduce conflicts without incurring the full cost of fully associative lookup.

- Since the total number of lines is fixed by cache capacity and line width, increasing the number of ways necessarily reduces the number of sets.



### Address Subdivision

> A real cache line contains both **data** and **metadata**. Besides the valid bit, the metadata includes the tag and other status. So a “16 KB cache” means 16 KB of payload data, not the total silicon storage consumed.

In practice, an address is split into three fields: `Address = Tag | Set selection | Byte offset`. The **offset** selects the specific byte inside a block; the set selector chooses the set; the tag identifies the block.

Useful formulas: if cache size is `C`, line size is `B`, and associativity is `A`:

1. Number of lines = `C / B`

2. Number of sets = `(C / B) / A`

3. Payload bits per line = `B × 8`

4. **Set bits** = `log2(#sets)`

5. **Offset bits** = `log2(B)`

6. **Tag bits** = `32 − set bits − offset bits` on a 32-bit address machine

> Why do more ways imply fewer set bits? Because with total capacity fixed, the total number of lines is fixed. More ways per set means fewer sets.

![](https://i.postimg.cc/RFrqWwK3/jie-ping2026-03-26-12-07-09.png)



### Block-Size Trade-Off

> Larger cache blocks should, in principle, reduce miss rate: if each miss brings in more nearby data, later accesses are more likely to hit.

The two main reasons are stronger **spatial locality** and reduced **tag overhead**.

⚠️ But larger blocks are not always better:

- Fewer total blocks means less ability to keep many different regions resident at once, hurting temporal locality.
- Overly large blocks may cause **cache pollution** or require too many transfer cycles on the bus.
- These effects may offset the theoretical reduction in miss rate.

Two practical balancing rules of thumb:

1. Small caches usually prefer moderate block sizes such as 32–64 B, while larger caches may benefit from 128–256 B blocks.
2. Even when blocks are large, techniques such as **early restart** and **critical-word-first** reduce the time the CPU actually waits by delivering the most urgently needed word first.

</br> 

## Hit and Miss, Read and Write

Misses can be divided into read misses and write misses.

### Cache Read

1. **Read hit**: the CPU continues normally.

2. **Read miss**: an I-cache miss affects instruction fetch, while a D-cache miss affects data access. The simplest response is a **blocking cache**, which stalls the pipeline while fetching the missing block from the next level.

### Cache Write

1. **Write hit**: the addressed block is already in the cache. Possible policies include:

   - **Write-through**: update both the cache and main memory at the same time. The advantage is that memory always stays current, so dirty data is less likely to be lost if the cache fails. The disadvantage is **increased memory traffic**, heavy pressure on the bus and DRAM, and a stretched WB stage. For example, if the base CPI is **1** and 10% of instructions trigger a 100-cycle memory write, the effective CPI becomes `1 + 0.1 × 100 = 11`.

     A common remedy is a **write buffer**: place the pending write into a buffer, let the CPU continue, and let the memory controller write to main memory in the background. Conceptually, the write buffer is a **FIFO queue**.

     However, a write buffer only hides the problem temporarily: if the store frequency exceeds the DRAM write throughput, the buffer fills up and the CPU must still stall.

     **Coalescing** — merging writes to the same or neighboring blocks — can reduce traffic somewhat, but write-back is still often preferred.

     ![](https://i.postimg.cc/Kc3fpP96/jie-ping2026-03-31-09-00-26.png)

   - **Write-back**: write only the cache copy immediately, and write to main memory later when replacement occurs.

     This requires a **1-bit** modification record called the **dirty bit**, which is part of the block metadata: 0 means the cache line matches memory, while 1 means the line has been modified and memory is stale. Only when a dirty line is evicted must the data be written back. This write-back can itself be buffered.

     ⚠️ The point of write-back is that it is enough for the correct value to exist in either the cache or main memory; the two need not remain identical at all times. But if another processor / core wants to read that same data, then a **cache-coherence protocol** such as MESI becomes necessary.

     Advantages: lower WB latency, improved throughput, and **lower memory traffic**, which is especially friendly to multiprocessors.

     Disadvantages: a cache failure risks data loss, so systems often rely on **ECC (Error Correcting Code)**, commonly on the order of 8 bits per 64 data bits. This increases the cost of maintaining correctness.

     > For example, a direct-mapped 16 KB RV32 cache with 64 B blocks contains 256 lines, 512 bits of actual data per line, 6 offset bits, 8 index bits, 18 tag bits, 1 valid bit, 1 dirty bit, and 64 bits of ECC. In total that is **596 bits per line**, so the whole cache occupies **152576 bits**, which is substantially larger than the nominal 16 KB of payload data.相比 16KB (131072 bits) 大了不少。

2. **Write miss** (the addressed block is **not** currently in the cache):

   - **Write-allocate (fetch on write)** naturally pairs with write-back: first fetch the block from the next level, then modify it. This is suitable when the data is likely to be read or written again soon, as with local-variable or array-element updates.

   - **No-write-allocate** often pairs with write-through: simply **bypass** the cache on a miss. This is suitable for large sequential writes such as `strcpy`, `memset`, `memcpy`, or array initialization, where the data may not be reused immediately.

⚠️ Modern caches commonly use **write-back + write-allocate**, rather than write-through with no-write-allocate.

### Non-blocking read miss

> Stall-on-Use (Non-blocking miss) **vs** Stall-on-Miss (Blocking miss)

If we do not want every miss to lock the cache completely, we use **Miss Status Holding Registers (MSHRs)** to record miss transactions — effectively buffering misses. An **MSHR file** entry typically stores the block address, target information, and a valid bit.

Advantages:

- Two typical capability levels are **hit-under-miss** (continue serving hits while a miss is outstanding) and **miss-under-miss** (track additional misses while another miss is still pending).

- **Lockup-free caches** support **memory-level parallelism (MLP)**, allowing multiple memory accesses to be waited on or processed in parallel.

### I-cache miss

Step 1: send the original PC (⚠️ usually the current PC − 4) to memory;

Step 2: let main memory or the next cache level fetch the entire **instruction** block;

Step 3: write the returned block into the I-cache, storing the **data**, the high-bit **tag**, and setting the **valid bit**;

Step 4: restart instruction fetch.

> **Prefetching** is a common optimization for reducing I-cache misses. However, predicting which code region a program will execute next is not always easy. **Hardware instruction prefetch** is common; explicit software-directed I-cache prefetch is much rarer.

</br> 

## Memory Support

How can we increase main-memory bandwidth and reduce effective miss cost?

![](https://i.postimg.cc/v81hJ9v7/jie-ping2026-03-31-11-44-07.png)

Strategy 1: an ordinary organization with uniform width, where the bus is also 32 bits wide and transfer is slow.

Strategy 2: widen the data path. In theory, making the bus wider is attractive, but the engineering cost is high — **pin limitations**, **signal-integrity challenges** (interference and timing skew), and severe routing complexity.

⚠️ Strategy 3: **interleaved memory banking**. Instead of making a single bus or memory chip extremely wide, split memory into multiple banks so consecutive addresses can be distributed among them and worked on in parallel (MLP). This is usually the best solution.

</br>

# Cache Performance

## CPU time with cache

CPU time consists of two parts: **normal execution time + memory-stall time**.

1. `CPU time = (Program execution cycles + memory stall cycles) × clock cycle time`

2. `Actual CPI = Base CPI + I-cache stall + D-cache stall`

- **Program execution cycles** already include **cache hit time**.

- **Memory stall cycles** are mainly caused by **cache misses**, whose penalties are one of the main reasons CPI grows.

  `Memory stall cycles = (Memory accesses / Program instructions) × Miss rate × Miss penalty`

### Example

Suppose a program has I-cache miss rate = **2%**, D-cache miss rate = **4%**, miss penalty = **100 cycles**, base CPI (ideal cache) = **2**, and load/store instructions account for **36%** of execution. What is the actual CPI?

- IF always requires memory access, so the instruction-side penalty is `0.02 × 100 = 2`; only 36% of instructions access data memory, so the data-side penalty is `0.36 × 0.04 × 100 = 1.44`. Thus the real CPI is the base value plus memory stall: `2 + 2 + 1.44 = 5.44`, meaning the program is slowed by a factor of 2.72.

> **If CPI is abnormally high, the culprit is often not slow arithmetic units but memory stall.**

</br> 

## AMAT

> **AMAT = Average Memory Access Time**. CPI cares about the whole program — **how many cycles each instruction costs on average** — while AMAT only cares about memory access — **how much time each access costs on average**, in **cycles/access**.

- ⚠️ `D-Cache AMAT = Hit time + Miss rate × Miss penalty`, so AMAT cannot be judged from miss rate alone — **hit time also matters**.

- ⚠️ `Overall AMAT = %instr × I-cache AMAT + %data × D-cache AMAT`, where `%instr` is the fraction of accesses going to instructions. Usually **%instr > %data**, though SIMD / vector-heavy programs may reverse that relationship.

</br> 

## Improving Cache Performance

Clearly, the AMAT formula tells us that cache performance can be improved by reducing **hit time**, reducing **miss rate**, or reducing **miss penalty**.

### 1. Lower Hit Latency

To reduce hit time, we want the CPU to spend as little time as possible on each cache lookup. Typical ideas include small caches, way prediction, **virtual-address caches / VIPT**, low associativity, physical proximity to the CPU, and parallelizing address translation with cache lookup.

</br> 

### 2. Decrease Conflict Miss

**Cache Miss —— 4C's**

1. **Compulsory / cold miss**: the first access to a line must fetch it from lower memory.
2. **Conflict / collision miss**: the set size (associativity) is insufficient to hold several lines that map to the same set.
3. **Capacity miss**: the cache itself is too small to hold the required working set.
4. **Coherency miss**: in a multiprocessor setting, invalidation requests from other processors may force the line out.

> A conflict miss occurs when main-memory lines with the **same set bits** are accessed near one another, but there are not enough ways to keep them all resident.

#### Strategy 1: Bigger Associativity

If conflict misses are frequent, one should not only think about enlarging the cache, but also about increasing associativity and redesigning the mapping behavior.

> Higher associativity means more complex MUX design, more comparators, and longer hit latency, but it also **reduces conflict misses**. ⚠️ A **2-way set-associative** design is often a good compromise.

- **Fully associative**: a memory block may be placed anywhere, and access searches the whole structure in parallel — requiring **one comparator per line**. This is flexible but expensive and slow.
- **N-way set associative**: map the block to a set by modulo, i.e. **`set No. = (block No.) % (#sets)`**, then search only within that set — requiring **N comparators**. This is a compromise between flexibility and cost.

- ⚠️ If **associativity increases**, then **the number of sets decreases** (since the total number of lines is fixed), so the index-bit count falls and the tag-bit count rises.

- In general, the higher the associativity, the lower the miss rate — but the benefit quickly shows **diminishing returns**, while **hit latency** and implementation cost grow significantly.

  For a 64 KB D-cache with 16-word blocks, representative SPEC2000 miss rates might look like this: 1-way **10.3%**, 2-way **8.6%**, 4-way **8.3%**, 8-way **8.1%**.

#### Strategy 2: Direct-Mapped + Victim Cache

For a direct-mapped cache, ⚠️ all main-memory blocks whose indices differ by multiples of the number of sets collide with one another and can cause conflict misses. By contrast, a **set-associative** design reduces conflict misses by giving each set additional buffer slots.

Patch 1: ⚠️ a **victim cache** is a tiny (often **4–8 line**) **fully associative** backup cache placed between a **direct-mapped L1 cache** and L2. It acts as a buffer for cache lines that were just evicted from L1 due to conflict misses.

Patch 2: **padding**. For example, if a 64-line cache repeatedly sees conflicts among arrays separated by 1024 elements, one may insert dummy gaps such as `A[1024], Dummy0[8], B[1024], Dummy1[8], C[1024]` to break the collision pattern. However, a compiler may optimize `Dummy[]` away if it is never actually used.

#### Comparing 2-Way Set Associative vs. Direct-Mapped + Victim Cache

- A **victim cache** is especially good at handling conflicts that are concentrated on a small number of addresses or a small number of sets and repeatedly evict one another.

  It exploits **high temporal locality** — the line was “just evicted, but will be used again very soon.”

- A **2-way** design is better when conflicts are not concentrated in a few hotspots, but are distributed across many sets, i.e. when temporal locality is weaker.

> In general-purpose systems (desktops, servers, laptops), 2-way set associativity is more common; direct-mapped + victim cache is more specialized and often appears in embedded or cost-driven designs.

#### Cache Block Replacement

(When a full set receives yet another conflicting line, how should the conflict miss be handled? Which resident line should be evicted?)

- In a direct-mapped design, there is no choice: with only 1 way per set, the existing line is simply overwritten.

- In set-associative or fully associative caches, one may choose among three classic policies: Random, **LRU (Least Recently Used)**, or **FIFO**.
  1. Random
  2. **LRU**: exploit **temporal locality** by evicting the block that has not been used for the longest time. **True LRU** requires stack-like tracking and is hardware-expensive; **pseudo-LRU** uses cheaper approximations, such as rotating pointers, but is less accurate.
  3. **FIFO**: ignore access history and evict blocks according to arrival order.

**General LRU**: “record the age of each cache block.”

- In a 4-block set, one can imagine a 2-bit counter per block tracking “how long since last use.” When one block is accessed, its counter resets to 0 while the others age upward. On a conflict miss, the block with the largest counter is replaced.

Theoretical Optimum：**Belady's Algorithm**

- From a pure **hit-rate** perspective, the ideal policy would evict the block whose next use lies farthest in the future.
- However: (1) this is impossible in reality because the future cannot be predicted exactly, and (2) ⚠️ it is not always optimal once **memory traffic** and **write-back cost** are taken into account, especially when dirty lines are expensive to evict.

> In practice, replacement policy must balance hit rate, clean/dirty status, memory traffic, power, and other engineering constraints.

⚠️ A weakness of **fully associative + LRU** is that certain cyclic access patterns can defeat it. For example, under a 5-block cycle `0,1,2,3,4,0,1,...`, a 4-line **DM(+VC)** setup may still achieve about **60% hit**, while a 4-line **FA+LRU** cache may repeatedly evict the exact block needed next, falling to **0% hit**.

</br> 

### 3. Smaller Miss Penalty

#### Strategy 1: Multilevel Caches

- This follows the memory-hierarchy principle: L1 sits closest to the CPU and is smallest and fastest; L2 serves as a larger, slower cache beneath it; some systems also add L3, often shared across cores; below that is main memory.

- Of course, consistency across multiple cache levels is nontrivial, especially when cores share data and write-back / dirty states are involved.
- For example, with two levels, an L1 miss may still hit in L2. Then the average memory access time becomes `AMAT = L1 hit time + L1 miss rate × (L2 hit time + L2 miss rate × L2 miss penalty)`, which captures the fact that many L1 misses are intercepted before reaching main memory.

- Typical hit rates might look like this: L1 **80–95%**, L2 **50–80%** (measured on L1 misses), L3 **20–40%**. Lower levels are larger, but they also face progressively harder miss cases.



#### Comparing One-Level and Two-Level Caches

Consider a 4 GHz CPU with base CPI = 1, miss rate = 2%, and memory latency = 100 ns.

- With only L1, the miss penalty is `100ns / (1/4)ns = 400 cycles`, so the actual CPI becomes `1 + 2% × 400 = 9`.

- Now add L2 with 5 ns access latency and 20% local miss rate. If the L1-to-L2 miss flow remains 2%, then the actual CPI becomes `1 + 2% × (5/(1/4) + 0.2 × 400) = 3`.
- That is a 3× improvement in effective CPI behavior.

⚠️ In a two-level hierarchy, L1 and L2 serve different design goals:

- For **L1**, the most important requirement is a very fast hit time. It may therefore be smaller and simpler, with stronger reliance on spatial locality. Some implementations use **write-through** from L1 into L2.
- For **L2**, the main goal is reducing miss rate and preventing access from falling to DRAM. Its hit time matters less than L1’s, and temporal locality is often more important. Because L2 is larger, it more often uses **write-back** to memory.



#### Other Techniques

1. As noted in the block-size discussion, **critical-word-first** and **early restart** reduce the effective stall time by delivering the urgently needed word before the rest of the line.
2. **Non-blocking caches** can use MSHRs to overlap multiple outstanding misses and exploit memory-level parallelism.
3. **Cache prefetching** does not eliminate misses; instead, it makes them happen earlier so that their penalties can be hidden.
4. **Write buffers** for write-back caches allow the CPU to continue while dirty blocks are written back in the background.
5. **Cache-aware code scheduling** can hide miss penalties at the compiler level.

</br> 

## Cache Coherence in Multiprocessors

In a single-core machine, cache is simply an acceleration layer between CPU and memory. In a multicore machine, however, **the same memory address may exist in multiple private caches at once, and those copies may differ**.

Write-back makes this problem more visible, but even write-through does not automatically solve it. So we need a **cache coherence protocol**.

![](https://i.postimg.cc/25hRPqTf/jie-ping2026-04-06-13-54-03.png)

### Snoopy Cache Coherence Scheme

> **All cache controllers “snoop” the coherence-related events broadcast on the shared bus.**

- If one processor wants to write a cache line, it must often **invalidate** the copies in other caches so that only one valid up-to-date copy remains.
- If another processor wants to read a line whose latest copy is dirty in some private cache, that cache may have to provide the data. In other words, a cache is not only a consumer; it can also become a data supplier.
- Examples of snoopy-coherence systems include SGI Challenge, Sun Enterprise, multiprocessor PCs, and similar machines.



**A cache controller receives inputs from both sides** — it must respond not only to the local processor, but also to coherence events broadcast by others on the bus.

- Requests coming from the local processor: this core wants to read, write, or handle a miss.
- Requests and responses coming from the bus / snooper: another core asks to read-share or write-exclusively, a block is invalidated, or a block must supply data.



In other words, the controller may update its state, provide data, or initiate a bus transaction (for example, requesting write-exclusive access).

> **The protocol is a distributed algorithm: a collection of cooperating state machines.**

- There is no central brain. Each cache controller maintains its own **state machine**, and the controllers coordinate through bus events to preserve coherence.

> **The granularity of coherence is a cache block.**

- Coherence is managed at the level of an entire cache line / block.



⚠️ The **MESI protocol** is one of the classic four-state cache-coherence protocols.

- **M = Modified**: the block exists only in this cache, has been modified, and main memory is stale.
- **E = Exclusive**: the block exists only in this cache but still matches memory; a write turns it into M.
- **S = Shared**: the block may appear in multiple caches, and all copies match memory; before writing, a core usually invalidates the other copies.
- **I = Invalid**: the local copy is invalid and must be fetched again before use.

- For example, one may see a MESI-style state evolution such as `I -(read)-> S -> E -(write)-> M -> I`.



⚠️ **Snoopy coherence is not very scalable**, because it relies on a **global bus**. In very large systems, broadcasting every coherence event to every core becomes a bottleneck. So snoopy protocols are best suited to small or medium-scale shared-bus systems.

</br> 

### Directory-Based Cache Coherence

For larger systems, a centralized or distributed **directory** records **which processors currently hold copies of a given cache line**.

- Instead of broadcasting to everyone, the system maintains a **directory**.
- The directory knows both the state of a block and which caches currently hold copies.



When a processor wants block X, it first consults the directory, which returns both the current state and the locations of any copies. The directory then selectively instructs the relevant nodes to invalidate or share.

> **A directory turns “broadcast to the whole company group” into “private messages to the relevant people.”**

- Advantages: better **scalability**, lower traffic, and more architectural flexibility.

- Disadvantages: higher complexity, potentially longer latency, and extra storage overhead for the directory itself.

</br> 

### Inclusive and Exclusive Caches

- In an **inclusive** hierarchy, anything in L1 must also be present in L2. So L2 can help track L1 contents. However, when L2 evicts a block, it must also invalidate the L1 copy — a **back invalidation**.

  Inclusive hierarchies often allow larger lower-level line sizes.

- In an **exclusive** hierarchy, blocks are not duplicated: if a block is in L1, it is not also kept in L2. Blocks evicted from L1 are moved down into L2; an L2 hit migrates the block upward.

  Exclusive hierarchies often keep the same line size at each level.

The advantage of inclusion is easier coherence management, since L2 can help track which L1 caches contain a given block.

The advantage of exclusion is better capacity utilization, since an inclusive L2 spends some space duplicating L1 contents.

> In practice, many modern processors are willing to sacrifice some capacity efficiency in exchange for simpler coherence management.

### Other Notable Facts

- On advanced CPUs, especially **out-of-order** ones, the cost of a cache miss is no longer just “how many cycles did the machine stop.” The processor may hide part of the miss penalty if there is enough independent work to do while waiting.

- Combined with non-blocking caches, multiple misses may overlap, yielding memory-level parallelism.

- The impact of a miss depends on the program’s data flow: a **critical-path miss** hurts badly, while a miss off the critical path may be mostly hidden. AMAT is only an approximation — **not all misses are equally expensive**.

- Misses are often **correlated**, not independent random events. This is one reason modern prefetchers increasingly invite ML/AI-style ideas.

- Low arithmetic complexity does not automatically imply high real performance. **Locality matters**: the way a program accesses data can dominate the actual running time.

  For example, when the data set becomes large, radix sort may require fewer primitive operations than quicksort, yet still fail to run faster because of cache-miss behavior.

- Again, **cache is not just the L1/L2/L3 structures inside a CPU; it is a general design idea.** For example, **CDNs such as Akamai are caches at the scale of the network**, and GPU memory systems also employ cache-like hierarchy ideas.
