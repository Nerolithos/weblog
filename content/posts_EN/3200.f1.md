# C++ Data Structures (Linear): Linked Lists, Stacks, Queues, Maps, and Sets

</br>

**Arrays**, **vectors**, and **linked lists** are among the more elementary C++ data structures and can be implemented rather directly from scratch. **Trees** and **maps**, by contrast, are more complex structures already supported in the standard library.

**Stacks** and **queues** are restricted-operation abstractions of linear tables (LIFO / FIFO), and can be implemented either with arrays or linked lists. A **set** may be implemented using several different foundational data structures.

Arrays were already explained earlier, so this post first emphasizes **linear data structures**. In the latter part we turn to the nonlinear structures of trees, graphs, and heaps.

</br>

</br>

# I. Linked Lists: “Array vs. Linked List” and Pointer Manipulation

</br>

## Abstract Definition

A **singly linked list** consists of **discrete** memory blocks connected in one direction by **pointers**: **each node stores a pointer to the next node**.

![](https://i.postimg.cc/G3vPxpMw/LLdrawio.png)

- **Goal**: store a sequence of data while supporting **insertion** and **deletion** more easily than arrays do, at the cost of traversal-based access.
- **Arrays and vectors**: **fast access, difficult insertion / deletion**. Because they use contiguous memory, they support O(1) random access, but middle insertion or deletion requires shifting elements and dynamic resizing is less direct.
- **Linked lists**: **fast insertion / deletion, slow access**. Each node is allocated separately and chained by `next`, so memory is non-contiguous. Insertion or deletion only changes pointers, often in O(1), but random access becomes O(n).

</br>

</br>

## Core Code Template

Node and list-body structure:
```cpp
template <typename T>
class Node {  // node
public:
    T element;  // data member: stores data of any type
    Node *next;  // data member: pointer to another node

    Node(const T& e = T(), Node *n = nullptr)  // class constructor
    // the node stores `element` and a pointer `next` to the next node
        : element(e), next(n) {}  
        // initialize the members; for example: new Node<int>(5, nullptr)
};

template <typename T>
class List {  // list body
private:
    Node<T> *dummyHead;  // dummy head node
public:
    List() {  // default constructor
        dummyHead = new Node<T>(T(), nullptr);
    }
    Node<T>* zeroth() { return dummyHead; }
    Node<T>* first()  { return dummyHead->next; }  // pointers access members through ->
    bool isEmpty() const { return first() == nullptr; }  // determine whether the list is empty
};
```

**Advantages of using a dummy head**:
- `dummyHead` is a `Node<T>` instance serving as the formal head of the list. Its data field is `T()` (typically an empty / zero-like value), and its pointer is initially null.
- It guarantees that a newly created list **always has a head node**, even when the actual list is empty, so later code can consistently rely on `dummyHead`.
- To test whether the list is empty, one only needs to check `dummyHead->next == nullptr`.

</br>

Review points:
- `Node<T> *nPtr = new Node<T>(...)` does **not** mean `Node<T>` is a pointer class; it means we create an instance and immediately store its pointer in `nPtr`.
- For pointer basics: if `T *ptr = &a`, then `*ptr` behaves like `a`, while `&a` behaves like `ptr`.
- `ptr->member` is equivalent to `(*ptr).member`, and **pointers access member variables and methods through `->`**.
- **Template classes** let the same code work with different data types:
```cpp
template <typename T>  // 或写 class T，等价
T add(T a, T b) { return a + b; }
auto x = add(1, 2);  // 类型自动推导 -> add<int>
auto y = add<double>(1, 2.5);  // 显式指明模板实参
// 以及在模板类外，调用模板：
template <typename T>
struct Stack { void push(const T&); };  // 类内成员函数不需重申 template
template <typename T>  // 在类外的成员函数必须再写一次 template
void Stack<T>::push(const T& x) { /*...*/ }
```

</br>

</br>

## Typical Member Functions

### Insert `data` after node `p`

![](https://i.postimg.cc/j5pwmBh9/jie-ping2025-11-21-14-43-49.png)

```cpp
template <typename T>
void List<T>::insert(const T& data, Node<T>* p) {
// p 是老节点的内存位置（插在哪）
    Node<T>* newNode = new Node<T>(data, p->next);
    // 此刻 新节点的 next 被初始化为“插入前 p 的 next 指针的原本值”。如果新节点插在 p、q 间，那么 newNode->next 指向 q；如果 p 后原本没有节点，就指向 nullptr
    p->next = newNode;  // 将老节点的 next 改指向这个新节点，完成插入
}
```
The basic linked-list structure is as follows (in practice, nodes are usually passed around directly by pointer, so `&` is not written everywhere):

`<head> DH(null, &N1)→N1(e1, &N2)→N2(e2, nullptr) <tail>`



Note:

1. The “create node” and “redirect pointer” steps cannot be reversed, or the chain will break. However, they can be **combined** into `p->next = new Node<T>(data, p->next);`. This works because assignment has lower precedence, so the right-hand side is evaluated first and still sees the old `p->next`.
2. `dummyHead` itself can also be used with this insertion function.
3. One must ensure that `p != nullptr` and that `p` truly belongs to this list.
4. Since the implementation uses raw pointers plus `new`, do not forget to traverse and `delete` every node in the list **destructor**.

</br>

### Find the node (pointer) whose value is `data`

```cpp
template <typename T>
Node<T>* List<T>::find(const T& data) {
    Node<T>* p = first();  // 指向 dummyHead 后的第一个实节点
    while (p) {  // 遍历，直到 nullptr (等价于 false) 断开
        if (p->element == data) return p;
        p = p->next;  // 改指向下一个节点
    }
    return nullptr;  // 没找到
}
```

</br>

### Delete the node with value `data` by first finding its **predecessor**

```cpp
template <typename T>
Node<T>* List<T>::findPrevious(const T& data) {  // 找前一项才能得到指针
    Node<T>* p = zeroth();
    while (p->next) {
        if (p->next->element == data) return p;
        // “当前项指向下一项的指针”访问下一项的数据，符合就传回这个“指向下一项的指针”
        p = p->next;
    }
    return nullptr;
}
template <typename T>
void List<T>::remove(const T& data) {
    Node<T>* p = findPrevious(data);
    if (p) {  // 之前没找到 data 就不会删任何东西，因为指针为空
        Node<T>* tmp = p->next;  // 指向 data 再后面一项
        p->next = tmp->next;  // 前一项跳过 data 指向再后面一项
        delete tmp;
    }
}
```
Note: strictly speaking, we are not “deleting the value itself”; we are reconnecting neighboring pointers and removing that node from the chain.

One should also implement:
- destructor `~List()`: delete all nodes starting from `zeroth()`
- copy constructor and assignment operator: perform a **deep copy** of each node rather than copying pointers only

</br>

</br>

## Complexity Compared with Arrays

Arrays:
- Access `a[i]`: O(1)
- Insert/delete at the head: O(n) (because everything must be shifted)

Singly linked list:
- Sequentially access the kth element: O(k)
- Given a pointer `p`, insertion / deletion after `p`: O(1)

A classic exam question is: **when should one use an array, and when a linked list?**

- Need lots of random access → array / vector
- Frequent middle insertion / deletion and uncertain length → linked list

</br>

</br>


## Things to watch for

1. In insertion / deletion, **connect before disconnecting** or **connect before deleting**, otherwise the chain may be lost.
2. The difference between a version **with a dummy head** and one without.
3. **Deep copy / destructor**: avoid memory leaks and shallow-copy problems.

# II. Stack: LIFO, Array Version, and Linked-List Version

</br>

## Abstract Definition

- Interface: `push(x)`, `pop()`, `peek()`, `isEmpty()`
- Property: last-in, first-out (LIFO). It behaves like a pile of cards: the last one placed on top is the first one removed. Operations occur at one end only (the top), and all basic operations are O(1).

</br>

</br>

## Stack Class Template

- Header file form:
```cpp
template <typename T>
class Stack {
public:
    Stack();
    bool isEmpty() const;
    void push(const T& x);
    T& pop();   // 删除并返回栈顶
    T& peak();  // 只读栈顶
};
```

</br>

</br>

## Array Implementation (Sequential Stack)

```cpp
const int MAX_STACK = 1000;  // 固定容量

template <typename T>
class Stack {
private:
    T items[MAX_STACK];  // “数组”栈
    int top;  // top == 当前占用位数 == 下次插入位置(当前栈顶以上一项)

public:
    Stack() : top(0) {}  // 初始化：top=0，数组为空
    bool isEmpty() const { return top == 0; }

    void push(const T& x) {
        if (top >= MAX_STACK)
            throw StackException("stack full");
        items[top++] = x;  // 先写入，再把 top 加 1
    }
    T& pop() {
        if (isEmpty())
            throw StackException("stack empty");
        return items[--top];
        // 先将 top-1 ，然后返回这个值(顶项值)。下一次push 会占用 top-1，盖掉这个原本的值
    const T& peek() {  // 不能改变栈
        if (isEmpty())
            throw StackException("stack empty");
        return items[top - 1];  // 返回顶项指
    }
};
```
The array-based stack has the following structure from bottom to top:

`<bottom> [i_0, i_1, ..., i_n] <top>,   top = n+1`

- Limitation: **capacity is fixed**, so overflow is possible; this can be improved by using a dynamic array such as `std::vector` (or manually resizing `new T[capacity]`).
- Recall: `++x` increments before use, while `x++` uses the old value before incrementing.

</br>

</br>

## Linked-List Implementation (Linked Stack)

Required members: a pointer `topPtr` to the stack top, and `next` pointers that point to older nodes.
```cpp
template <typename T>
class StackNode {  // 节点类
public:
    T item;
    StackNode *next;
    StackNode(const T& e = T(), StackNode* n = nullptr)
        : item(e), next(n) {}
};

template <typename T>
class Stack {
private:
    StackNode<T> *topPtr;
public:
    Stack() : topPtr(nullptr) {}  // 初始化：topPtr指向空，链表为空
    bool isEmpty() const { return topPtr == nullptr; }

    void push(const T& x) {
        auto *newPtr = new StackNode<T>(x, topPtr);  // 自动推导指针类型
        topPtr = newPtr;  // 下一个节点会指向现在的节点：后进先出
    }
    const T& peek() {
        if (isEmpty()) throw StackException("empty");
        return topPtr->item;
    } 
    T& pop() {
        T& stackTop = T();
        if (!isEmpty()) {
	        stackTop = topPtr->item;
	        StackNode<T> *tmp = topPtr;
	        topPtr = topPtr->next;
	        delete tmp;  // 将顶项的指针移出了链表
	    }
		return stackTop;
	}
};
```
- A linked stack is like a reversed linked list: the `next` pointer points toward older nodes. From bottom to top, the structure looks as follows (`N` denotes a `StackNode`):

  `<bottom> N1(nullptr, i1)←N2(&N1, i2)←N3(&N2, i3)←N4(&N3, i4)←...`

![](https://i.postimg.cc/dQGRj0Ww/jie-ping2025-11-21-13-44-51.png)

</br>

</br>


## Things to watch for

Typical uses of stacks:
- **Parenthesis / HTML-tag matching**, and **expression evaluation** (operator stacks and operand stacks).
- **Backtracking** algorithms (Sudoku, permutations, state rollback in path search).
In short, stacks are natural for undo-like and rollback-like problems.

# III. Queue: FIFO, Circular Array, and Linked-List Version

</br>

## Abstract Definition

- Interface: `enqueue(x)` insert at the back, `dequeue()` remove from the front, `getFront()` inspect the front, `isEmpty()` test emptiness.
- Property: first-in, first-out (FIFO). Like a pipe, what enters first also leaves first. It is common in waiting lines, task scheduling, and buffers. Unlike a stack, it uses both ends; like a stack, its basic operations are O(1).

</br>

</br>

## Linked-List Implementation (with `front` / `back` pointers)

Required members: pointers to the front and back nodes for insertion and removal, with `next` pointing toward newer nodes.
``` cpp
template <typename T>
class QueueNode {
public:
    T item;
    QueueNode *next;
    QueueNode(const T& e = T(), QueueNode* n = nullptr)
        : item(e), next(n) {}
};
template <typename T>
class Queue {
private:
    QueueNode<T> *frontPtr;  // 双向检索
    QueueNode<T> *backPtr;
public:
    Queue() : frontPtr(nullptr), backPtr(nullptr) {}
    bool isEmpty() const { return frontPtr == nullptr; }

    void enqueue(const T& x) {
        auto* newPtr = new QueueNode<T>(x, nullptr);  // 指向新节点
        if (isEmpty()) frontPtr = newPtr;
        else backPtr->next = newPtr;  // 原本队尾的 next 指向新节点
        backPtr = newPtr;  // 队尾变成这个新节点
    }
	void dequeue() {
	    if (isEmpty()) throw QueueException("empty");
	    auto* old = frontPtr;              // 旧队头
	    frontPtr = frontPtr->next;         // 前移指针，将旧队头移出队列范围
	    if (!frontPtr) backPtr = nullptr;  // 若队列被删空，把尾指针也置空
	    delete old;                        // 释放旧队头
	}
    void getFront(T& x) const {
        if (isEmpty()) throw QueueException("empty");
        x = frontPtr->item;
    }
};
```
- All basic operations are O(1), much like a stack.

- Unlike a linked stack, a linked queue is a forward linked list: the `next` pointer points toward newer nodes (toward the tail). From head to tail the structure looks like this (`N` denotes a `QueueNode`):

  `<head> N1(&N2, i1)→N2(&N3, i2)→N3(&N4, i3)→N4(nullptr, i4) <tail>`

![](https://i.postimg.cc/zDTwFfd5/jie-ping2025-11-21-13-46-05.png)

</br>

</br>

## Circular-Array Implementation (Sequential Queue)

The idea is to connect the two ends of a finite array into a ring. Then, even though capacity is fixed, deletion at the front no longer requires shifting all elements; once an index runs out of range, it simply wraps back to `0` (the start of the array). The **front position is therefore not fixed**.

![](https://i.postimg.cc/xd4r551N/jie-ping2025-11-21-13-53-10.png)

```cpp
const int MAX_QUEUE = 1000;

template <typename T>
class Queue {
private:
    T items[MAX_QUEUE];
    int front;  // 队头下标，注意不一定是 0
    int back;   // 队尾下标
    int count;  // 当前元素数
public:
    Queue() : front(0), back(MAX_QUEUE-1), count(0) {}
    bool isEmpty() const { return count == 0; }

    void enqueue(const T& x) {
        if (count == MAX_QUEUE) throw QueueException("full");
        back = (back + 1) % MAX_QUEUE;  // *核心逻辑*
        // back 一旦超出 MAX_QUEUE 就从 0 开始回填，保证索引不溢出 (上一行的 throw Exception 确保这种“回填”绝对不会覆盖掉 front，因为超出 MAX_QUEUE 了)
        items[back] = x;
        ++count;
    }
    void dequeue() {
        if (isEmpty())
            throw QueueException("empty");
        front = (front + 1) % MAX_QUEUE;  // 同理，防止索引向负数溢出
        --count;
    }
    T& getFront() const {
        if (isEmpty())
            throw QueueException("empty");
        return items[front];
    }
};
```
Note: in an array of length 1000, valid indices run from 0 to 999. After `dequeue`, the former first element is simply outside the queue range, not physically moved away. To prevent overflow when `enqueue` reaches 1000, modular arithmetic with `MAX_QUEUE` creates a **circular array**, so `1000 ≡ 0`, `1001 ≡ 1`, and so on. This avoids shifting all elements forward after each `dequeue`, which is why `back` may very well appear before `front`.

</br>

</br>


## Things to watch for

1. **Circular-array index update formula**:
```cpp
   back = (back + 1) % MAX_QUEUE; ++count;    //  enqueue 扩容
   front = (front + 1) % MAX_QUEUE; --count;  // dequeue 减容
```
2. Comparing linked-list queues and array queues:

   Linked lists do not require fixed capacity.

   Array implementations must distinguish “full” from “empty,” but usually have smaller constant factors.

3. Queues are commonly used in operating-system task queues, network-packet buffers, print queues, and similar settings.

# IV. Maps: Key–Value Pairs, Hash Functions, and Buckets

</br>

## Abstract Definition

- A map (dictionary-like structure) maintains a set of `(key, value)` pairs. It resembles a mathematical mapping relation between two sets, though it **does not have to be bijective** (i.e. it need not be a perfect one-to-one correspondence).
- Typical operations: `operator[]` (can both insert and search), `put(key, value)` (insert or update), `get(key)` (find the value by key), `containsKey(key)`, `remove(key)`, `size()`, and `isEmpty()`.

The standard C++ library already provides two implementations:
- `std::map<K,V>`: based on a balanced BST (red-black tree), ordered, O(log n).
- `std::unordered_map<K,V>`: based on a hash table, unordered, average O(1).

</br>

</br>

## Vector (Dynamic Array) Implementation: Linear Search / Binary Search

A simple way to implement a `string → string` mapping dictionary:
```cpp
struct KeyValuePair {
    std::string key;
    std::string value;  // 键值对 {key, value}
};

class StringMap {
public:
    StringMap() {}
    ~StringMap() {}
    // 构造/析构都没自定义逻辑，vector 类会自己管理元素生命周期
    std::string get(const std::string& key) const {
        int index = findKey(key);  // 见 private 域，获取索引
        return (index == -1) ? "" : bindings[index].value;
        // 找不到则返回空字符串
    }
    void put(const std::string& key, const std::string& value) {
        int index = findKey(key);
        if (index == -1) {  //  如果当前不存在这个 key
            KeyValuePair kv{key, value};
            // 这是一个在第一行声明的键值对实例，被打包塞进 bindings
            bindings.push_back(kv);  // vector 类的成员方法（插入）
        } else {
            bindings[index].value = value;  // 存在就覆盖原本的值
        }
    }
private:
    std::vector<KeyValuePair> bindings;
    // 用动态数组 bindings 存储键值对 {key, value}
    int findKey(const std::string& k) const {
        for (int i = 0; i < (int)bindings.size(); ++i)
            if (bindings[i].key == k) return i;  // 遍历返回对应索引
        return -1;  // 找不到，返回异常状态码
    }
};
```
- `std::vector` is essentially a dynamic array: use `.size()` for the current size, `.push_back(e)` to append an element, and `.value` / `.key` to access the stored value or key.

- Structure of the `bindings` dynamic array:

  `std::vector [{key1, value1}, {key2, value2}, ... , {key-value pair}]`

- This implementation is **not surjective** in the mathematical sense, because `key` values may not repeat, so one key cannot map to multiple values. If `put()` is called with an existing key, it overwrites the old value.

- This implementation **can still be injective** only in the loose everyday sense that different keys may share the same value (**values may repeat**).

- Search and insertion are both O(n), so this is very inefficient and used only for demonstration.

- Recall: the code above uses the **ternary operator** for case-based handling.
```cpp
return (index == -1) ? "" : bindings[index].value;
// 核心结构：<cond ? expr1 : expr2>，逻辑：cond 真就传回 expr1；假就传回 expr2
```

</br>

</br>

## Hash-Table (`unordered_map`) Implementation

A hash table is the standard-library method that implements a map through a **hash function**.

Key idea:

1. The core is the **hash function `hashCode(key)`**: map a key to a nonnegative integer; then take modulo to obtain the bucket index, i.e. `index = hashCode(key) % nBuckets;`. If the hash function is poor, the table degenerates.
2. **Buckets**: an array whose slots each represent a bucket, typically along with a pointer to the first node in that bucket.
3. **Linked list / chain**: each node stores `{key, value, next}`; nodes in the same bucket are connected via `next`.
4. Each bucket stores a **linked list** (or vector-like chain) to resolve **collisions** through separate chaining.



Three components of a hash table:

1. A hash function `hashCode(key)`: map the key into an integer that is **as collision-free as possible**.
2. A bucket array: a fixed-length array of size `B`, where each slot is a “bucket.”
3. **Collision handling**: what to do when two different keys are mapped to the same bucket. `unordered_map` uses **separate chaining**, so each bucket stores a small linked structure.

</br>

Private fields of a hash table:
```cpp
class StringMap {
private:
    struct Cell {
        std::string key;
        std::string value;  // 键值对
        Cell *link;  // 链指针，指向下个键值对节点
    };
    Cell **buckets;  // 指向桶数组
    int nBuckets;    // 桶个数
    int count;       // 已存 key 数量
    Cell* findCell(int bucket, const std::string& key);
};
```



The hash-table structure is roughly as follows:

```
桶数组 → (键, 值,  next指针)
[0]   →  ("AR", "5", ptr) → ("AK", ... ) → null
[1]   →  ("AL", ... ) → null
[2]   →  null
...
[nBuckets]   →  ("NV", ... ) → ...
```

</br>

</br>

## Instantiating `std::unordered_map`

```cpp
#include <unordered_map>
#include <string>
#include <iostream>

int main(){
    std::unordered_map<std::string, int> m;  // 实例化“字符串→整数”哈希表
    m.reserve(1024);              // 预留 1024 个键值对，减少 rehash 频率
    m.max_load_factor(0.7f);      // 限制装载因子，限制桶内链表长度

    m.emplace("alice", 1);   // 等价于 insert({k, v})，不会覆盖存在的键值对
    m["bob"] = 2;  // 这是操作符：operator[](k) = v，会覆盖存在的键值对

    if (auto it = m.find("alice"); it != m.end())
        std::cout << it->first << " -> " << it->second << "\n";

    std::size_t bc = m.bucket_count();
    std::size_t b  = m.bucket("alice");  // "alice" 键被哈希函数放到了哪个桶？
    std::cout << "bucket_count=" << bc
              << ", bucket(alice)=" << b
              << ", bucket_size=" << m.bucket_size(b) << "\n";
}
```
 Hash-table key-value structure: `unordered_map<first, second>`

Bucket size and rehashing:

- `.size()` gives the total number of key-value pairs across all buckets; `.bucket_size(i)` gives the number inside bucket `i`; `.bucket_count()` gives the total number of buckets; `.reserve(e)` reserves room for about `e` entries.
- The load factor is $\frac{.size()}{.bucket\_count()}$, meaning **the average number of key-value pairs per bucket**. `.max_load_factor()` sets its upper bound; once the table grows too dense, a **rehash** is triggered, expanding the bucket array and redistributing all keys.
**Whether inserting or searching, one first uses the hash function to determine which bucket the object should belong to.**
Inserting items:
- Without overwriting: `emplace(k, v)` or `insert({k, v})`
- With overwriting: use `operator[](k) = v`, for example `map[k] = v`
- To search for an item: `.find(k)` searches by key in average O(1). If found, it returns an iterator to that element; otherwise it equals `.end()`.

</br>

</br>

## Comparing Map Implementations

1. **Balanced BST (`std::map`)**: (we discuss concrete operations later)
    - Ordered (can be traversed in increasing key order)
    - Operation complexity: O(log n)
2. **Hash table (`std::unordered_map`)**:
    - Unordered; iteration order is unstable
    - Average O(1), worst-case O(n) (when the load factor grows too high and many collisions appear)
    - The key issues are hash-function quality and load-factor control
3. A teaching-oriented implementation using **vector + linear / binary search**:
    - Ordinary performance (O(n))

</br>

</br>


## Things to watch for

1. **Understand the structure of a hash table**:

   The hash function maps a key to an integer bucket index → assign a bucket → place the item into the chain / structure inside that bucket.

   Why are collisions unavoidable? Because imperfect hash functions and a finite number of buckets force different keys to map to the same bucket.

2. **Load factor and rehash**:

   Why does performance degrade when the load factor grows? Because bucket chains become longer, causing lookup and insertion to deteriorate.

# V. Sets: Set Operations and Implementations

</br>

## Abstract Definition

A set is an unordered, non-repeating enumerative data structure. If the same data is inserted into an array `A` and a set `S`, the difference reveals the total number of duplicates.

`A.size() - S.size()`

Set interfaces:

- `add(x)` / `insert(x)`: add an element; `remove(x)`: delete an element; `contains(x)`: test membership;
- and mathematical operations such as ∪ (union), ∩ (intersection), and − (difference).

</br>

</br>

## Comparing Four Implementation Strategies

You mainly need to master the “linked-set” implementation.

</br>

### 1. Pointer / Linked-List Implementation
   - This is similar to a linked list: one may use a dummy head, let each node represent one element, find the predecessor first, and then remove the following node by changing its `next` pointer. The difference is that **before insertion one must call `contains()` to avoid duplicates.**
   - Time complexity: search, insertion, and deletion are average O(1).
      The following shows a head-insertion implementation of `contains()` + `insert()`:
```cpp
// 对于 Node {T val; Node* next;}
	bool contains(const T& x) const {
	    for(auto* p = dummyHead->next; p; p=p->next) {  // head 代表占位空节点
	        if(p->val==x) return true;
	    }
	    return false;
    }
    void insert(const T& x) {if(!contains(x)) dummyHead->next = new Node<T>(x, dummyHead->next);}  // 先查重
    // 先将 head->next 赋给 new Node，然后把 new Node 的指针赋给 head->next
```
The structure of such a “linked set” from the head looks like this:

`DH(null, &N1) → <insertion position> → N1(v1, &N2) → N2(v2, nullptr)`

**Advantages**: simple foundational implementation; good enough for **very small** sets (tens of elements); useful for duplicate checking.

**Disadvantages**: high time complexity, slow lookup, pointer overhead, and poor cache friendliness.

</br>


### 2. BST Implementation (`std::set` in the standard library):
- In C++, BST-based sets are generally implemented as **red-black trees**.
- The key operations `insert` / `erase` / `find` are O(log n) on average and in the worst case.
- Supports **ordered traversal** (inorder traversal). We explain the concrete implementation later in the tree section.

</br>


### 3. Hash-Based Implementation (`std::unordered_set`):
A **hash set** is the standard-library method of implementing a set with a **hash function**.
- A hash function maps keys to buckets; collisions inside a bucket are handled with chains or equivalent bucket structures. Average insertion / deletion / lookup is O(1), with worst-case O(n) if the load factor grows too high and hashing degenerates.
- No ordering is guaranteed; performance depends on a good hash function and rehashing to control the load factor.
Below is an example of instantiating a hash set to **store points**:
```cpp
#include <unordered_set>
struct Point { int x,y; };
struct PointHash {  // 哈希函数
    size_t operator()(const Point& p) const noexcept {
    // 把 x,y 混合成一个 64 位数，也就是把 Point 映射成 size_t
        return std::hash<long long>()(( (long long)p.x<<32 ) ^ (unsigned)p.y);
    }
};
struct PointEq {  // 相等比较函数
    bool operator()(const Point& a, const Point& b) const noexcept {
        return a.x==b.x && a.y==b.y;
    }
};
std::unordered_set<Point, PointHash, PointEq> us;
// unordered_set 要求进入其中的类型传入”元素、哈希函数、比较相等函数“
us.reserve(1024);  // 预留 1024 个点位，减少 rehash
us.insert(Point{1,2});
bool has = us.find(Point{1,2}) != us.end();  
// 判断找到的指针不等于空，返回 true
```
A hash set needs three components:
1. A custom stored element type
2. A good hash function
3. A way to test whether two elements are equal

**Advantages**: in most scenarios, this gives the **fastest membership query**; it is ideal when ordering does not matter.

**Disadvantages**: unordered; sensitive to hash-function quality; rehashing invalidates iterators; memory fragmentation may occur.

**When to choose it**: mainly when the dataset is large and the primary task is fast **membership testing**, not ordering.

</br>

### 4. Bit-Vector Sets (`std::bitset`) and Set Operations:

- When the element universe is **finite and indexable** (for example ASCII characters or small-range integers), one can use **1 bit per element** to indicate membership: 1 means present, 0 means absent. The full row of bits is called a **characteristic vector** or **bit vector**. Then set operations reduce to **integer bitwise operations** — union: `A | B`, intersection: `A & B`, difference: `A & ~B`. This is extremely fast for small integers and character sets. Note that **bitwise reading proceeds from right to left**.

![](https://i.postimg.cc/vmLhHHpV/jie-ping2025-11-21-14-08-12.png)

- Single-element operations are O(1); bulk union / intersection / difference take **O(n / word_size)**.
  
  Below is an example of instantiating a bit-vector set for set operations, using `std::bitset<N>` over `[0, N-1]`:

```cpp
#include <bitset>
std::bitset<1000> A, B;  // std::bitset 不可以在之后运算时扩容
A.set(3); A.set(10);
bool has3 = A.test(3);
A.reset(10);
auto C = A & B;        // 交集
auto D = A | B;        // 并集
auto E = A & ~B;       // 差集
auto F = A ^ B         // 异或
```
Bit operations:
- Set to 1: `.set(index)` changes bit `index` to 1, thereby **including** that element; without arguments it sets the whole set to 1.
- Set to 0: `.reset(index)` changes bit `index` to 0, thereby **excluding** that element; without arguments it resets the whole set to 0.
- Index operator `operator[](index)`: `bitSet[index] = bin` assigns the `index`th bit directly.
- Counting 1s: `.count()` computes how many elements the set contains.

</br>

Below is an example of implementing a bit-vector set from scratch:
```cpp
const int BITS_PER_BYTE = 8;
const int BITS_PER_LONG = BITS_PER_BYTE * sizeof(long);
const int requiredWords = (RANGE_SIZE + BITS_PER_LONG - 1) / BITS_PER_LONG;  // ceil(R/W) = (R+W-1)/W 是向上取整的标准形式
// requiredWords：大小为 RANGE_SIZE 的集合需要多少长整数来储存其特征向量？
struct CharacteristicVector {
   unsigned long words[requiredWords];
   // 定义”无符号长整数型“数组 words 及其容量
};
unsigned long createMask(int k) {  
    //  第 k 位掩码：只有第 k 位是 1。与它进行 AND 操作会只剩下第 k “向量维度”
   return unsigned long(1) << k % BITS_PER_LONG;
   // 强制转化 1 为长整型，然后左移 ”k 在所属 long 中的位数“，创造掩码
}
bool testBit(CharacteristicVector & cv, int k) {
    if (k < 0 || k >= RANGE_SIZE) {
       error("testBit: Bit index is out of range");
    }
    return cv.words[k / BITS_PER_LONG] & createMask(k);
    // 如果整个数组的第 k bit 为 1 就返回真
}
void setBit(CharacteristicVector & cv, int k) {
   if (k < 0 || k >= RANGE_SIZE) {
      error("setBit: Bit index is out of range");
   }
   cv.words[k / BITS_PER_LONG] |= createMask(k);
   // 将第 k bit 所在的那个 long 替换成它 OR 第 k 位掩码，也就是把它变 1
}
void clearBit(CharacteristicVector & cv, int k) {
   if (k < 0 || k >= RANGE_SIZE) {
      error("setBit: Bit index is out of range");
   }
   cv.words[k / BITS_PER_LONG] &= ~createMask(k);
   // 将第 k bit 所在的那个 long 替换成它 AND 第 k 位掩码的否定，也就是把它变 0
}
```
- The width of `unsigned long` is 32 or 64 bits depending on the machine. Recall that `long(...)` is just a cast to type `long`; it does not meaningfully apply to character strings.
- Recall the common integer formulas for ceiling and floor:
- The core concept here is the **mask**: a value used with AND to suppress irrelevant bits.

![](https://i.postimg.cc/nLFxLMCN/jie-ping2025-11-21-14-19-30.png)

- In bitwise calculation, **numbers are read from right to left** in significance: for example, `long(3)` is `000...011`, and `long(3) << 2` becomes `000...01100`; similarly, “set bit 5” means `long(1) << 5`.

- `k / BITS_PER_LONG` tells you **which `long` contains bit `k`**, while `k % BITS_PER_LONG` tells you **which bit position inside that `long`**.

- All the `long` elements in `words[]` together represent the **characteristic vector of one set**, and each of them contains `BITS_PER_LONG` bits of binary information.

  

**Advantages**: **extremely fast and extremely memory-efficient** (1 bit per element); batch operations are efficient; well suited for **character sets, dense small-integer sets**, and graph-algorithm “visited” markers.

**Disadvantages**: the universe size must be **known and not too large**; `std::bitset<N>` requires `N` to be a **compile-time constant**; dynamic ranges require `vector<uint64_t>` or `boost::dynamic_bitset`.

**When to choose it**: when elements are **small-range integers** (for example within `0..10^6`) and many set operations are needed, or when extremely fast membership testing and batch union / intersection are required.
