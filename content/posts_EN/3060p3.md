# Code Scheduling



> The topic of this project is **code scheduling**: reorder instructions **without changing program semantics** so that the total execution time becomes shorter. The goal is to **hide the waiting time of long-latency instructions as much as possible** and **prioritize instructions on the critical path**, while respecting all dependency constraints.

</br>

Code scheduling trades a small amount of compile-time effort for a substantial runtime improvement. Speaking of compilation...

![](https://i.postimg.cc/Hkky6Pm7/compilecircleoflife-v0-zk74efewqv6f1.webp)

</br>

- Scheduling process: parse the instructions, build the dependency graph, compute priorities, and produce the final schedule.
- Why it matters: different instructions have different latencies. For example, `lw` and `mul` are usually slower than `add` and `jal`. If we simply wait in program order, we create idle cycles (bubbles). A good scheduler **fills those waiting periods with other independent instructions whenever possible**.

</br>

## Static Latency Model

We assume that each instruction class has a **fixed latency**, and the scheduler uses that latency to estimate when a result becomes available. In real hardware, the latency of the same instruction is not always constant, but the SLM provides a simple and stable approximation that is good enough to guide dependency analysis and reordering.

![](https://i.postimg.cc/nLH1tC0F/jie-ping2026-03-14-11-00-05.png)

The latencies in the SLM are clearly “average-case” estimates. Real latency fluctuations do not hurt its usefulness too much. First, one may think of “latency” in the same way one thinks of **weights** in A* or Dijkstra-style shortest-path algorithms: it only needs to reflect relative cost. Second, if the estimate captures the broad trend — the relative time costs among instructions — then, much like Amdahl’s Law, optimization can already produce a large speedup. Pursuing exact precision would reduce generality and quickly run into diminishing returns.

- Multiplication can be decomposed into “generate partial products + add them level by level,” so it often takes 3–4 cycles. Division, by contrast, repeatedly compares, subtracts, shifts in a new bit, and adjusts the quotient bit by bit, so it is much slower, often taking tens of cycles.
- A `load` instruction may take as little as 5 cycles on a cache hit, while a `store` is always 1 cycle in this model. This is because `load` writes a register and therefore creates a RAW dependency: later instructions that read that register must wait for the value to arrive from memory. A `store` does not modify a register, so it creates no such dependency and need not be waited on.

</br>

</br>

## Dependency DAG

The dependency graph treats each instruction as a node and uses directed edges to represent dependencies or ordering constraints between instructions. For example, `I0 → I3` means that `I3` depends on `I0`, so `I0` cannot be scheduled after `I3`.

> Why must it be a Directed Acyclic Graph?

Because dependencies encode a **topological ordering**. If the graph contained a cycle, such as `A → B`, `B → C`, `C → A`, then no instruction in the cycle could be executed first, and scheduling would be impossible.

Which edges need to be present can be determined by analyzing register dependencies among instructions.

</br>

### Data Dependency: RAW (Read After Write)

RAW means that a later instruction needs to read a register value written by an earlier instruction. This is a **true dependency**: the successor really must wait until the predecessor has produced the value. RAW is also the only kind of dependency here that carries an actual latency rather than merely an ordering constraint.

For example:

```assembly
I0: lw  x1, 0(x5)
I1: add x3, x1, x2
```

- `I1` reads `x1`, and `x1` is written by `I0`, so `I1` has a RAW dependency on `I0`. Therefore the start time of `I1` must be later than “start time of `I0` + 5 cycles.”
- In implementation, one often maintains a `LastWriter[reg]` structure so that RAW edges can be added whenever a later instruction reads a register.

</br>

### Anti-Dependency: WAR (Write After Read)

WAR means that the earlier instruction reads a register, while a later instruction writes the same register. This is an **anti-dependency**: informally, the earlier read depends on the later write occurring sufficiently late. WAR does not require a specific waiting time; it only imposes an **order-only dependency**.

For example:

```assembly
I0: add x3, x1, x2
I1: li  x1, 0
```

- If `I1` were moved before `I0`, then `I0` would no longer read the original `x1`. So `I1` must start later than `I0`.

- A common implementation maintains some `LastReaders[reg]` structure. When a writing instruction is encountered, WAR edges are added from **all** earlier readers that still matter.

</br>

### Output Dependency: WAW (Write After Write)

WAW means that two instructions both write to the same register, and their write order must be preserved. This is also an anti-dependency, more specifically an **output dependency**. Again, it does not require an explicit delay, only an ordering constraint.

For example:

```assembly
I0: li x1, 5
I1: li x1, 9
```

- The final program should preserve the result of the last write. If the order is reversed, the final register value is wrong, so `I1` must start later than `I0`.

- In implementation, one usually also uses `LastWriter[reg]` to add WAW edges to later writers of the same register.

</br>

So, to summarize, drawing the DAG means doing the following.  
(Bonus: these can often be done in a single pass.)

- **RAW**: the current instruction wants to **read**, so find the most recent earlier **write** and add a RAW edge.
- **WAW**: the current instruction wants to **write**, so find the most recent earlier **write** and add a WAW edge.
- **WAR**: the current instruction wants to **write**, so find **all** relevant earlier reads and add WAR edges to **all** of them.

> Why must WAR look for all prior readers?

→ Because we do not know which earlier reader may move around under reordering. So WAR typically requires a **list of readers**, not merely a single “last reader.”

> Why do WAR and WAW not need latency values?

→ Because WAR and WAW enforce only safe overwrite ordering. They do not require the later instruction to consume a value produced by the earlier one. Once the execution order is fixed, no extra waiting time is needed.

> Why is there no RAR (Read After Read)?

→ Because two reads do not modify the register contents. Swapping their order does not affect correctness, so there is no need for such dependency edges.

![](https://i.postimg.cc/t4wY3gmm/jie-ping2026-03-14-12-09-26.png)

⚠️ Note: if an instruction both reads and writes the same register, such as `add x1, x1, x2`, it must **not** be detected as depending on itself. Otherwise we would create a self-loop and the scheduler would deadlock.

</br>

</br>

## *Simplifying Assumptions of the Project

To keep the project pedagogically clean, it uses a heavily simplified execution model:

- all dependencies are based only on **register dependencies**
- **no memory aliasing**
- single-threaded execution
- no control flow / exceptions / side effects
- memory operations are atomic

Under this model:

> Registers are the only shared state, so correctness constraints are exactly the RAW/WAR/WAW relations.

In the real world, things are more complicated: `store` / `load` may depend on one another through aliased memory addresses; in multithreaded settings the order of `lock` / `release` matters critically; looking only at registers may miss real constraints. Without these simplifying assumptions, a register-only DAG is not enough.

</br>

</br>

## Priority: Critical Path Length

> Why use priority, and how is it computed?

When multiple nodes are ready at the same time, the scheduler must decide which one to issue first. In this project, we use **Critical Path Length** as the priority.

**CPL (Priority)** = the **longest path length** from the current node to any exit node.

The reason is straightforward: if the critical path below a node is longer, then more dependent work is waiting behind it. Scheduling that node earlier helps unlock later instructions sooner and can reduce the overall completion time.

![](https://i.postimg.cc/85dw69wz/jie-ping2026-03-14-14-11-56.png)

Only RAW carries a real latency requirement, so CPL is **not** computed by blindly adding latency along every edge:

- for **RAW successors**, add the current node’s latency;
- for **WAR/WAW successors**, keep only the ordering relation and do **not** add the current latency.

</br>

</br>

## List Scheduling

This is the core algorithmic component of the project, and one of the most classical scheduling algorithms.

- **ready set**: all nodes whose predecessor constraints have already been satisfied and can therefore begin now.
- **inflight set**: nodes that have already started but have not finished yet, usually stored in a form like `(node, finish_cycle)`.

### Algorithm Flow

1. Initialize `cycle = 0`
2. Put every node with `unscheduled_predecessors = 0` into `ready`
3. On each cycle:
   - if `ready` is not empty, select the best node and issue it
   - add it to `inflight` and record its finish cycle
   - for WAR/WAW successors: **release them immediately upon issue**
4. Increment `cycle`
5. Check which nodes in `inflight` retire during this cycle
   - for the RAW successors of those nodes: **release them upon finish**
6. Repeat until all nodes have been scheduled

Again, the crucial point is **when** a successor is released:

- A **RAW successor** can only be released when the predecessor **finishes**, because it genuinely needs the data produced by the predecessor.
- A **WAR/WAW successor** can be released as soon as the predecessor **starts**, because those edges enforce only ordering, not value availability.

</br>

</br>

## Complete Example

Consider the following five instructions:

```assembly
1 I0: lw x1 , 0( x5)     ; latency =5, writes x1
2 I1: lw x2 , 4( x5)     ; latency =5, writes x2
3 I2: add x3 , x6 , x7   ; latency =1, independent
4 I3: mul x4 , x1 , x2   ; latency =3, reads x1 , x2
5 I4: sw x4 , 8( x5)     ; latency =1, reads x4
```

Analysis: first come two independent `lw` instructions, then one independent `add`, then a `mul` that depends on both loads, and finally a `sw` that depends on the `mul`.

So the DAG is:
- `I0 → I3`
- `I1 → I3`
- `I3 → I4`
- `I2` is independent

![](https://i.postimg.cc/CLMJk7R7/jie-ping2026-03-14-14-59-31.png)

Now compute CPL:
- `priority(I4) = 1`
- `priority(I3) = 3 + 1 = 4`
- `priority(I0) = 5 + 4 = 9`
- `priority(I1) = 5 + 4 = 9`

⚠️ However, if we also consider resource constraints, our scheduler is **single-issue**, meaning at most one instruction can begin per cycle. So `I0` and `I1` cannot start in the same cycle; one of them must start in the second cycle. Therefore the lower bound on the completion time becomes:

`max(finish_time(I0), finish_time(I1)) + latency(I3) + latency(I4) = 6 + 3 + 1 = 10`

So the shortest possible finishing time is 10 cycles.

A valid scheduling order is:
`{I0, I1, I2, I3, I4}`

Although the final issue order happens to match the original program order, the scheduler inserts the independent `I2` during the waiting period for the loads, thereby hiding two idle cycles. The documentation explicitly describes this as **filling idle slots** and **hiding latency**.

> Scheduling optimization does not have to mean a dramatic change in instruction order. Its real purpose is to use idle gaps intelligently.

</br>

</br>

## Tie-Breaking Policy: How to Choose When Priorities Are Equal

When several ready nodes have the same priority, we still need a secondary rule to decide which one to issue first. The project uses a multi-level comparator in `selectBestNode()` with three strategies:

1. **SMALLER_INDEX**: choose the node with the smaller original program index.  
   Advantage: this stays closer to the original order and acts as a conservative tie-breaker.

2. **MOST_CHILD**: choose the node with more successors.  
   Advantage: if a node can unlock more later instructions, issuing it first may make more nodes ready earlier, improving the chance of filling latency bubbles.

3. **LPT (Longest Processing Time)**: choose the node with the largest latency.  
   Advantage: long-latency instructions should start earlier, because subsequent instructions may then cover that latency bubble. When both short- and long-latency instructions are ready, and the later work can overlap with the long latency, LPT often beats index order.

> Compare **MOST_CHILD** and **LPT** first; if they are still tied, use **SMALLER_INDEX** as the final tie-breaker.

</br>

## Complexity Analysis

(Assume there are `N` instruction nodes and `E` dependency edges.)

**1. Naive list scheduling**: if each round scans all nodes to find the ready set, the complexity is relatively high:
`O(N^2 + E)`

**2. Recursive memoized priority calculation**: if CPL is computed by “recursion + memoization,” then each node and edge is effectively processed only a limited number of times, so the complexity is approximately:
`O(N + E)`
