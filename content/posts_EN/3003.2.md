# InfoVis Design Principles

**InfoVis**: interactive visual representation of abstract data to amplify cognition

- Purpose: insight, not pictures
- Best for: EDA (**Exploratory Data Analysis**)
- Use EDA when you do not yet know exactly what you are looking for.
- **Two aspects**: representation / interaction
- **Two challenges**: scale / diversity

</br>

### Edward Tufte's Principles

- **Graphical Integrity**: tell the truth. Check baseline, scale, and context. Avoid distortion or ambiguity; graphical magnitude should stay proportional to data magnitude.
   **Zero tolerance rule**: ordinary bar charts should begin from a zero baseline, or else visual exaggeration occurs. Dot plots are different because viewers are not comparing bar lengths there.
   **Lie Factor** = size of effect in graph / size of effect in data. A figure whose lie factor exceeds **1.05** is already meaningfully distorted.
   When using **area** rather than length (for example, circles), make sure the **area**, not the diameter, is proportional to the data.
- **Tufte Data-Ink Ratio**: DIR = Data_Ink / Total_Ink, need to MAXIMIZE.
   Methods: erase non-data ink, erase redundant data ink, then revise and edit.
   In some cases — especially when exact values matter, there is too much data, or the comparisons are weak — a table may be better than a chart.
- **Avoid chart junk**: unnecessary 3D, unnecessary color, decorative gridlines, boxes, overly dense labels, and gratuitous shapes.
- **Data Density** = Number of Entries / Area of Graph
- **Small multiples**: same design, another variable changes
- **Macro/Micro**: big picture + fine details
- **Content is King**

</br>

### Gestalt Laws

- **proximity**: when grouping objects, humans usually prioritize spatial closeness over color or shape
- similarity
- connectedness
- **continuity**: people tend to follow partially broken paths as though they were continuous
- **closure**: the mind automatically completes incomplete shapes
- **symmetry**: humans are highly sensitive to symmetric structures
- **relative size**: larger objects are often perceived as closer or more important, which may mislead viewers
- figure-ground
- **Prägnanz**: less is more

</br>

# Visual Perception

## Perception

- active / contextual, influenced by values, experience and background

**Two-stage model**: 

- **Early stage**: low-level, parallel perception (color, shape, etc.), corresponding to fast and intuitive automatic processing.
- **Later stage**: sequential, top-down, goal-directed processing (pattern, meaning, etc.), which is slower and relies on longer-term knowledge; it may be described in terms of verbal and motor subsystems.

**Three-stage model**: 

- **Selection**: what is noticed first; **Organization**: how it is grouped; **Interpretation**: how it is understood.

</br>

### Preattentive Processing

Fast parallel visual processing that does not require focused attention and usually completes within **200–250 ms**.

- Tasks: target detection / boundary detection / counting
- Only a **single feature** can usually be processed preattentively.
- A **conjunction** of features, such as shape plus hue, usually requires a sequential search.
- **Key properties**: color / form / movement / spatial position
- **Change blindness**: people sometimes fail to notice visual changes at all.

</br>

### Color Scheme

Color refers to light roughly in the **0.4–0.7 μm** range. Expert observers may distinguish on the order of a million colors, while ordinary viewers distinguish perhaps around twenty thousand.

- **RGB** (based on the three cone-cell channels) is common for screens, while **CMYK** is common in printing and physical publication.
- **HSL**: Hue, Saturation, Lightness.
   `L = 0` is black, `L = 100` is white, and **`L = 50` with `S = 100` gives a vivid pure color**.
- **HSV**: Hue, Saturation, Value
   `V = 0` is black, **`V = 100` gives full-value color**, and `S = 0` yields white / gray-scale appearance.
- **Luminance**: foreground and background should differ in brightness.
- **Quantitative color**: avoid rainbow scale for quantitative data
   Ware's rule of thumb: about **12 colors**.
   Less is more: **more than about 20–30 colors usually hurts rather than helps**.
   Practical advice: blue can occupy large areas if not oversaturated; avoid using nearby blue hues that are hard to separate; red and green should be used carefully, especially away from the focal center; related hues may be varied through saturation and lightness.

</br>

# D3.js and HTML

The standard stack of web visualization includes:

1. **HTML** — the markup language of the page
2. **DOM** — the manipulable structural representation of the HTML page
3. **CSS** — the presentation layer: color, font, size, layout, and so on
4. **SVG** — a vector-graphics system that scales cleanly
5. **JavaScript / D3** — the main layer for storing, transforming, and dynamically rendering data

</br>

### CSS

- **Internal CSS**: directly style elements such as `h1`, `p`, and `body` inside the `<style>` area of the HTML file.
- **External CSS**: reference a separate stylesheet via `<link rel="stylesheet" href="xxx.css">`.

</br>

### JS

- **`console.log()`**: print output for debugging.
- **`object.key` is equivalent to `object["key"]`**: object properties can be accessed by either dot notation or square-bracket notation.
   ⚠️ Important exception: if the property name begins with a special character or a digit, or when a variable is used as the key, square brackets are required.
- Single vs. double quotes are largely interchangeable in many contexts, and semicolons are often optional in practice.
- Variables declared without `var`, `let`, or `const` become **global**, which is usually dangerous.
- `[x, y, z]` denotes an array, while `{a: b, c: d}` denotes an object / dictionary.
- **`this`** usually refers to the current object inside a method.
- **`const`** means a constant binding, not necessarily an immutable value.
   ⚠️ Thus one **can** still modify array elements or object properties, but **cannot** reassign the whole array / object reference.

</br>

### JS Arrays

`[]`: arrays are, in JavaScript terms, also objects.

- Objects use string-like keys; arrays use numeric indices starting from 0.
- **`push()`**: append to the end of an array.
- **`pop()`**: return the last item and remove it.
- **`shift()`**: return the first item and remove it.
- **`unshift()`**: add one or more elements to the front of the array and return the new length.
- **`slice(start, end)`**: a shallow copy of the interval `[start, end)`.
   ⚠️ It returns a **shallow copy** of part of the array while leaving the original unchanged.
- **`map()`**: transform every element through a function and return a new array, e.g. `map1 = arr.map(x => x + 2)`.
   ⚠️ **When not to use map()**:not using the returned array; or not returning a value from callback
- **forEach()**: calls a function once for each array element
   ⚠️ Unlike `map()`, this kind of traversal is about executing over elements, not about constructing a new array.

</br>

### D3

JavaScript library, data-driven DOM / HTML / SVG manipulation

- **Selection**: based on CSS selectors
   **select()** first matching; **selectAll()** all matching
   select by ID ("#xxx"), by className (".xxx"), by tag ("div", "h1", "p", "span", etc.)
   The return value of most selection methods is the selection itself. This means that selection methods such as .style, .attr and .on can be **chain**ed. E.g.: `selection.attr(...).style(...).text(...)`
- **`append()`**: append a child element.
- **`text()`**: set text.
- To change HTML content, use **`html()`**.
- To change attributes, use **`attr()`**.
- To change CSS styling, use **`style()`**.

</br>

### SVG

Scalable vector graphics `<svg>`: the drawing canvas for SVG graphics. **Create with**: `d3.select("#vis").append("svg")`.

- **circle**: `cx`, `cy`, `r`. Optional attributes include **`fill`**, **`stroke`**, and **`stroke-width`**. **Create with**: `.append("circle")`.
- **rect**: `x`, `y`, `width`, `height` (the last two cannot be negative).
- **line**: start point `x1, y1`; end point `x2, y2`.
- path: **Capital letters**: absolute position；**Lowercase letters**: relative position
- **text**: x, y, text-anchor
- **g**: use `vis.append("g")`, then append shapes such as `group.append("circle")`, `group.append("rect")`, and `group.append("text")` inside that group.
- Updating and controlling displayed elements:
   **d** = datum; **i** = index. **Use i for position (where), Use d for size / style (what/how)**
   ⚠️ **data join** follows four steps: `d3.select() → .selectAll() → .data() → .join()`.
   Once the data changes, one typically performs the join again so that `update(newData)` can redraw / update the visualization.
   **`.join(enter, update, exit)`** uses three functions to handle element entry and update when data exists, and element exit when it no longer does.
- Element transformation tools:
   **translate(tx, ty)**: move along x and y
   **rotate(angle)**: rotate around origin or specified center
   **scale(sx, sy)**: resize, if sy omitted then sy = sx
   **Linear scale**: domain → range
   **`scaleSqrt`**: useful for circle-area encoding, so that **area**, rather than diameter, stays proportional to data.
   **`axis`**: `axisBottom`, `axisLeft`, `axisRight`, `axisTop`, used via `.call()` to generate and invoke the axis.
   **transition**: state change animation

</br>

</br>

# Multivariate Data & Tables

### Data

- Variables in data are attributes or dimensions.
- **Data model**: often described in terms of **objects**, **attributes**, and **relations**.
- In a data table, a **case** is the “individual” represented by a row, while a **variable** is the “attribute” represented by a column.
   Wide table: each attribute gets a column (e.g. year)
   A **long table** stores each case as an attribute-value pair; it is often tidier and easier to use for plotting.
   Variable Type: 
  1. **Categorical**: **nominal** (not ordered, e.g. sex, country tag) or **ordinal** (ordered, e.g. clothing size).
  2. **Quantitative** (numeric and measurable).
      **Metadata**: data that describes data itself (such as YAML / TOML headers).
- Strategies for encoding data and improving space usage include composition, alignment, folding, recursion, and overloading.

</br>

### Data Issues and Cleaning：

- Common data issues include **missing fields**, **duplicates**, **conflicts**, delimiter problems, whitespace, encoding problems, abbreviations, typos, syntax errors, and so on.
- Data cleaning: **remove, impute (guess) or flag (e.g. NAN) missing ones**. Only remove an outlier if you can prove that it’s **erroneous**.
- Typical cleaning steps: remove unwanted or irrelevant observations, fix structural errors, standardize units / formatting / capitalization, remove unwanted outliers, fix contradictions, fix type-conversion and syntax errors, deal with missing data, and validate the results.lidate dataset.

</br>

### Multivariate Data, Table and Graph:

- **Univariate** data graph: x-axis: independent cases, y-axis: dependent variables
- **Bivariate**: Scatter-point is most often
- **Trivariate** and **Hypervariate** scatter graph: **use small multiples, or colors in 2D**
- A **table** is best for exact values. Numbers are usually right-aligned, often also aligned by decimal position, which improves comparison.
- **Graph** = patterns / shape / relationships。

</br>

</br>

# Multivariate Graphs & Charts

**Few’s process**: Choose chart (or table) based on data type and analytic goal, not decoration.

</br>

### Basic Plots:
 **Scatter plot**: well suited to showing the relationship between **two** quantitative variables, but weaker for many-variable relationships.

- If time intervals are irregular, a scatterplot may be more appropriate than a line chart.
   **Line chart**: used to show change over continuous time.
- As long as it does not mislead, the y-axis does not always have to start from 0.
- **Banking to 45°**
- If there are too many overlapping lines (**overplotting**), use **small multiples**.
- When values span different orders of magnitude and you want to compare **growth rates** (for example, all growing by 10%), a **log scale** is better than a linear scale.
   **Bar chart**: because the comparison is based on **length**, starting the y-axis above 0 can severely exaggerate differences.
- **Horizontal bars** are often better than vertical bar charts when **labels are too long**, **there are too many cases**, or **the bars themselves would become awkwardly long**.
- **Stacked bars**: allow multiple category components to be layered inside one bar.
   **Trellis display**: essentially a form of small multiples. For example, it can replace an overly cluttered stacked bar chart.
   **Crosstab**: suitable for multivariable cross-analysis among categorical variables, especially beyond a 2×2 case.
   **Pareto chart**: essentially a combination of a **bar chart + line chart**.
- ⚠️ **80/20 Rule**: 80% of effect comes from 20% of causes (**Vital few**)。
- It is useful for analyzing the root causes of problems or defects, such as complaints, sales problems, or accident analysis.
   **Bump chart**: shows how rankings change over time.

</br>

### Distribution Analysis Plots:
 **Box plot**: used to examine distribution, spread, and possible outliers.
 **Histogram**: used for continuous data; its bars touch and show a distribution. ⚠️ A bar chart, by contrast, is for discrete data and has separated bars.

- Overlaying multiple datasets may become cluttered; **a line chart can sometimes handle that more clearly**.
   **Dot plot**: shows the frequency with which some values appear in a dataset, using dots — somewhat like a point-based histogram.
- ⚠️ Dot-style categorical comparisons are mainly for one variable or a few grouped comparisons (primarily 1D), whereas a scatter plot is inherently pairwise and two-dimensional.
   **Strip plot**: a scatterplot with categorical grouping along one axis, displaying individual observations for each category.
   **Stem-and-leaf plot**: displays the shape and spread of a continuous distribution, revealing **central tendency, variability, skewness, and outliers**.

</br>

### Cyclical Data Plots:
 **Radar graph**: represent data across the cyclical nature of time

- Suitable for time structures with periodicity.
   **Heatmap**: display a large quantity of cyclical data
- ⚠️ When periodic data is too dense for a radar chart, a heatmap may be more suitable.

**Overplotting**: many data points overlap, making interpretation difficult.

- Possible remedies: reduce marker size, **jitter points**, make points transparent, aggregate or sample the data, or **encode density**.

</br>

# Graph theory and graph visualization

### Graph theory

A graph consists of nodes (entities) and edges (relationships), i.e. `G = (V, E)`. [For more graph-theory detail, click here](https://blog.nero-lithos.com/posts/3200.f2/).
Two common graph representations:

- **Adjacency Matrix**: supports random access, with complexity `O(|V|^2)`; entry `(a, b) = 1` indicates that edge `ab` exists.

- **Adjacency List**: supports sequential graph traversal, with complexity `O(|E|)`; a list entry such as `a: b` indicates that edge `ab` exists.

  Nodes are often assigned numeric indices for easier management and programming; numeric IDs can be generated through maps (Java), dictionaries (Python), or even SQLite.
  ⚠️ Graphs are useful in business for detecting risk patterns, simulating internet / railway / supply-chain systems, optimizing asset mixes in markets, mapping social hierarchies, and detecting communities.

- A graph need not be simple; for example, multiple-edge-like relationship structures may arise in social-network settings such as *LinkedIn*.

</br>

### Graph in Visualization

Applications of graphs and corresponding **graph layouts**:

- (Node-link) **Relationships** → nodes + links, conceptual picture
- **Hierarchies** → tree layouts, sunburst charts.
- **Communities** → clustered layouts, labels, computational clustering
- **Flows** → Sankey, chord diagram
- **Spatial networks** → schematic layout for routes/nodes
- **Large graphs** → overplotting or tile-based aggregation
- **Process of visualizing graphs**:
   **1. Prepare Data**
   **2. Layout**: arrange nodes and edges.
   **3. Add Visual Attributes**: size, color, position, and edge annotation.
   **4. ⚠️ Interact, Explain, Further Analysis**: zoom and pan, identify by hover / click, **filter** (⚠️ not deletion, only hiding), isolate, inspect neighbors and paths, **drag / move / modify / delete / group**, build explanatory sequences (e.g. in *Gephi*), annotate, add a **legend**, and export.

- ⚠️ Two *Gephi* filters: **giant component** (topologically remove tiny or disconnected components) and **edge weight** (remove low-frequency edges).
- **Ego-network**: one focal node together with all of its direct neighbors and relationships.

</br> 

Basic graph concepts:

- ⚠️ **Basic graph statistics** include **density**, **components**, **degree**, **path**, and **centrality**. (Hue does not belong here.)

  **Centrality** includes **betweenness centrality** (number of shortest paths going through a node), **Katz centrality** (weighted influence based on distance to all nodes), **degree centrality** (larger degree means more central), and **closeness centrality** (average distance to all nodes).

  **Diameter**: the longest among all shortest paths (i.e. the critical path).

- ⚠️ **Essential visual attributes**: Node size, Node color, Label, Edge weight, Edge color, Edge type.

</br>

Layout strategies:

Page-level arrangement:

- **Random node layouts**: nodes are randomly distributed in space, as in a rough Force Atlas–style layout.
- **Force-directed layout**: often yields one large cluster plus scattered peripheral points.

Graph-internal layout types:

- Force-directed, node-only, time-oriented, top-down, radial, treemap (nested rectangles whose areas express hierarchy), hierarchical pie (**a circular hierarchy, very similar to a sunburst**), map-based, chord, and parallel-coordinate layouts.
- ⚠️ **Hierarchy layouts** include both ordinary **tree** layouts and **radial** layouts. Tree layouts are non-circular and neatly arranged, such as top-down or left-to-right; top-down layouts can become too wide when labels are long. If the hierarchy is broad (many nodes), a radial layout may work better. Radial layouts include both radial trees and sunburst charts.
- A **Sankey diagram** is suitable for directional, step-based, path-like flow (for example, product-supply flow), whereas a **Chord diagram** suits many-to-many flow without a single fixed directional sequence (for example, global imports and exports).

</br>

Graph **encoding**: **Node** = entity, **Link** = relationship, **Size** often encodes importance or popularity, **Color** often encodes frequency or attribute, and **Width** often encodes link amount or strength.

- In community visualization, a **clique** is a complete subgraph (all internal pairs are connected), and color can be used to distinguish major cliques.

  A **convex hull** is the enclosing “shell” drawn around each group (community); it is useful for emphasizing groups rather than individual links.

- **Flow visualization** is used to display transactional events, depict systems or behavioral models, and represent supply chains.

  ⚠️ **Transactional data is often visualized with a Sankey diagram**: a multi-step flow where link **width encodes volume**.

</br>
