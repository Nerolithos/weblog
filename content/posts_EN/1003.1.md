## The First Java Program

```java
public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, world!");
    }
}
```

`println()` prints a line and then moves to the next line automatically, whereas `print()` does not append a newline.

**A Java source file (that is, a program) may define only one `public` class, and the class name must exactly match the file name.** The method `public static void main(String[] args)` is the fixed entry point of a Java program, so execution always starts from `main`. Here, `void` is the method return type and means “no return value”; `static` indicates that the method belongs to the class itself.

Use `javac` to compile a `.java` source file into `.class` bytecode, and use `java` to run a compiled Java program. The argument to `java` is the class name.

`args` is **not** a reserved word. You could rename it to `lgu` if you wanted, as long as you use the same name consistently: `lgu[0]`, `lgu[1]`, and so on. Index `[0]` refers to the first command-line argument. In the following example, if you type `java Hello lgu xys` in the terminal, Java will output `Hello! xys`.

```java
public class Hello {
    public static void main(String[] lgu) {
        System.out.println("Hello! " + lgu[1]);
    }
}
```

The full workflow is:

1. Define a `public` class named `Name`, and save it as `Name.java`
2. Run `javac Name.java` to compile it and generate `Name.class`
3. Run `java Name` to execute the class



## Data Types

Java defines the following primitive data types:

- Integer types: `byte`, `short`, `int`, `long`
- Floating-point types: `float`, `double`
- Character/string-related types: `char`, `String` (`String` is a reference type)
- Boolean type: `boolean`

The smallest storage unit in computer memory is the **byte**. One byte consists of 8 binary digits, i.e. 8 bits. Its binary range is `00000000` to `11111111`, which corresponds to decimal `0` to `255`, or hexadecimal `00` to `ff`.

Among the basic numeric types, `byte` occupies 1 byte, `short` 2 bytes, `int` 4 bytes, `long` 8 bytes, `float` 4 bytes, `double` 8 bytes, and `char` 2 bytes.

For booleans, the situation is more subtle:

**In the Java Virtual Machine specification**, boolean values are often treated as `int` values after compilation, and `int` occupies **4 bytes**. In that sense, a boolean may be regarded as taking 4 bytes. This is related to **memory alignment**: the JVM often aligns data to natural boundaries such as 4 bytes, which can improve performance and simplify implementation.

**Inside arrays**, however, each boolean element is typically represented using 1 bit in the abstract sense: Java encodes `true` as 1 and `false` as 0. So when booleans are stored as array elements, their **effective information content** is 1 bit each.

In short, although boolean values are always conceptually represented by 0 and 1, and the actual information needed is only 1 bit, the physical storage cost depends on context.

- The **IEEE 754** standard is used to represent `float` values (4 bytes, 32 bits):
  • First express a floating-point number in the form `(-1)^a x b x 2^c`
  • **Sign bit (1 bit, a)**: indicates whether the number is positive or negative  
   0 means positive, 1 means negative
  • **Exponent field (8 bits, c)**: stores the exponent using a bias  
   The bias is 127, so the actual exponent is stored exponent − 127
  • **Fraction / mantissa field (23 bits, b)**: stores the fractional part of the significand  
   The leading 1 is implicit and therefore not stored explicitly
  • For `double`, the bit allocation is 1, 11, and 52 respectively.

- For `float`, one concrete example is:
  `6.5 = (-1)^0 x 1.625 x 2^2 = 0 10000001 10100000000000000000000`
  • **Sign bit = 0** (positive number)  
  • **Exponent = 10000001 (129)**, so the actual exponent is `129 - 127 = 2`  
  • **Mantissa = 1.101<sub>BIN</sub>** (with the leading 1 implicit), so the significand is `1 + 0.5 + 0.125 = 1.625<sub>DEC</sub>`



In Java, integers are represented exactly, and integer arithmetic is also exact within range (with truncation toward zero in division). Floating-point numbers, however, often cannot be represented exactly in binary, so rounding errors may arise:

```java
//...
float a = 0.1 / 3 * 3
float b = 0.1
System.out.println(a == b)
//...
```

```shell
(terminal output) > False
```





**For an integer type occupying `n` bytes, the representable range is `[-2^(8n-1), 2^(8n-1)-1]`.**

- Why is it not `[-2^n, 2^n-1]`?

Computer systems use **two’s complement** to represent signed integers. For example, `+6<sub>DEC</sub> = 00000110<sub>BIN</sub>`, while to represent a negative number you invert the bits of its absolute value and then add 1: `-6<sub>DEC</sub> = 11111010<sub>BIN</sub>`. The leading bit indicates the sign, in a way analogous to the sign bit in floating-point representation. This is also closely related to overflow, which we discuss below.



**Type casting:**

- `byte b = 42, ~~b = (int) b~~`: a variable can have only one type; it cannot “recast itself” this way.
- `byte b = 42, ~~long c  = (int) b~~`: inconsistent target type — should it be `long` or `int`?
- `byte b = 42, int c = (int) b`: valid.
- Type casting can also lead to overflow if the target type cannot represent the value.



## Rules of Computation



Let us use an example from the 2024 TUT session 1:

```java
public class rules {
  public static void main(String[] args) {
    int a = 7 / 5;
    double b = 7 / 5
    double c = 7.0 / 5
    double d = 4.00 + 5.00
    System.out.print(a + ", " + b + ", " + c + ", " + d);
  }
}
```

```shell
(terminal output) > 1, 1.0, 1.4, 9.0
```



- In Java, the JVM does not treat all numeric literals uniformly as Python might. **When assigning a value to a variable, you must declare its type**, e.g. `int x = 42` assigns the integer 42 to `x`. Numeric literals themselves also have default types: for instance, `42` is an `int`, and `42.0` is a `double` (**all decimal literals are `double` by default**). So in the example above, for `b`, we have `7<int> / 5<int> = 1<int>`, and then `1<int>` is widened to `1.0<double>` when assigned to `b`. Since `7 / 5` is integer division, the result must be an integer. The same idea explains `c`: because `7.0` is a `double`, the result is `1.4<double>`. **If one operand is an integer and the other is floating-point, the integer is automatically promoted to floating-point.**
- An unspecified floating-point literal such as `42.0` is `double` by default in Java. If you want a `float`, you must append `f` or `F`, e.g. `float a = 42.0f`. Likewise, a `long` literal such as 42 should be written as `42L` when needed.
- In Java, `7.00 / 5` evaluates to `1.4` rather than `1.40` because, by default, Java prints floating-point numbers using only the necessary decimal places. In other words, it suppresses meaningless trailing zeros. For example, for `d`, the expression **`4.00 + 5.00` evaluates to `9.0` (a `double`)**.

- On **overflow in integer arithmetic**:

  **Integer overflow** is a characteristic pitfall of fixed-width arithmetic in Java: if the result exceeds the representable range of the data type, the computation wraps around and becomes incorrect.

  Here is an example cited from Professor Li’s lecture:

  ```java
  //......
  byte m = 100;
  byte n = 70;
  byte p = m + n
  System.out.print(p)
  //......
  ```

  ```shell
  (terminal output) > -86
  ```

  Assigning 170 to the `byte` variable `p` causes overflow, because the maximum value of `byte` is 127, and 170 exceeds that bound. Java wraps the value according to the `byte` range (`-128` to `127`). Since the total range size is 256, the result is effectively `170 - 256 = -86`. Intuitively, once you go beyond one end of the range, you re-enter from the other side.

- Bitwise operators (binary level):

  AND: `&`, OR: `|`, XOR: `^`.
