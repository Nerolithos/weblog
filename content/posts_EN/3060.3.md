# Week 5  Calling Convention and Its Registers

A calling convention includes: arguments, return values / return address, argument passing, stack entry and exit, frame-pointer setup and restoration, and which registers are preserved by the caller or callee.

## The Convention of Function Calls

### From the machine’s macroscopic perspective, what does a function / procedure call look like?

Starting from the caller...

1. **Prepare arguments** (if registers are insufficient, or the convention requires stack passing, place them on the stack)
2. **Save the return address** (`ret`)
3. **Jump to the callee**

Control is then transferred to the callee...

4. **Create a stack frame**: move the stack pointer downward, <u>reserving space for the caller’s frame, local variables, and saved registers</u> (`sp -= stackFrame`)
5. **Save the registers that must be preserved**
6. **Execute the function body**
7. **Prepare the return value** (possibly in registers `a0/a1`, or on the stack)
8. **Restore the saved registers** and **return to the caller**

- ⚠️ A **stack frame** is the private region allocated on the stack for a function; each function gets its own frame segment, usually growing downward. The **frame pointer `fp`** is the addressing base of the current function’s frame, and is used here to <u>restore the stack to exactly what it looked like before the call</u>.

</br>

### Meanwhile, what happens to the stack itself?

1. In steps 1–3 (on the caller side), the caller **creates its own stack frame** and <u>pushes arguments (if `a0`–`a7` are exhausted) and the return address onto the stack</u>.
2. In steps 4–7 (on the callee side), the callee **creates a new stack frame** and <u>stores local variables, saved registers, and dynamic objects inside that frame</u>.
3. After step 8 (back at the caller), **the callee’s stack frame is popped**, and the stack returns to its pre-call state, with `fp/sp` restored to their original positions.

### "Saved Regs"

The ABI specifies that some registers must be cooperatively preserved through stack saves and restores. This prevents variable clobbering and lifetime errors during chains of function calls. Conceptually, it is a bit like this:

*“The office computers belong to the IT department (the caller), but every machine has accounts from both IT and PR. When the PR department (the callee) borrows a machine to help IT, it must switch the account back to IT before returning it (callee-saved); meanwhile, IT is responsible for archiving its own important files before lending the machine out (caller-saved).”*

**A) Callee-saved (must be preserved by the callee)**

- `s0`–`s11` (saved registers), the old **`fp`** — in practice `s0` (i.e. `x8`) often serves as `fp` — and other **CSEs**.

Rule: **If function B wants to use or modify these registers, it must save the old values first and restore them before returning.**

→ These backups stored by the callee on the stack are the typical **saved regs**, and they usually have relatively long lifetimes.

**B) Caller-saved (must be preserved by the caller if needed)**

- `t0`–`t6` (temporaries), `a0`–`a7` (arguments), and **`ra`** (return address).

Rule: **Function B may freely overwrite them.** If function A still needs them after the call, A must save them in advance.

→ These values usually have relatively short lifetimes.
