# Week 8  Pipelined Implementation



> The core idea of **pipelining** is to **overlap different stages of multiple instructions**. An ideal pipeline is **single-issue**: it begins **one instruction per short cycle**.

Basic concepts:

- **Throughput**: on average, how many cycles are needed per completed instruction?
- **Latency**: how long does one instruction take from issue to completion?
- **Stage**: instructions are divided into hardware-distinct stages.
- **Fill**: instructions enter the pipeline one by one; **Drain**: we wait for the last instruction to pass through the remaining stages.



⚠️ **The RISC-V ISA is naturally pipeline-friendly**:

- **Fixed 32-bit instruction length** (except for the C-extension), which makes it easier to divide instructions into regular stages. By contrast, x86 instructions vary from 1 to 17 bytes.
- **Few and regular instruction formats**: decode and register-read can often be completed in the same stage.
- **Load/store architecture**: only `load` and `store` access memory; address calculation stays in EX and memory access stays in MEM, which cleanly separates responsibilities.

</br>

## Five-Stage Pipeline, Throughput, and Timing

In the classical 5-stage pipeline, each instruction is split into five stages that occupy different parts of the processor:

1. **IF**: Instruction Fetch, including the PC, the `ADD4` unit, and instruction memory.
2. **ID**: Instruction Decode and Register Read, including the register file and immediate generator (sign-extension).
3. **EX**: Execute / Address Calculation, including the ALU and its input selectors.
4. **MEM**: Data-memory access.
5. **WB**: Write Back to the register file.

In a single-cycle CPU, the minimum cycle length is the <u>total latency of the slowest instruction, typically `lw`</u> — a “long cycle.” In a multi-cycle or pipelined CPU, the minimum cycle length is the <u>latency of the slowest stage</u>, such as MEM or EX — a “short cycle.” The reciprocal of the cycle time gives the clock frequency.

![](https://i.postimg.cc/7L9T17R1/jie-ping2026-03-17-11-23-35.png)

- Suppose `1 long cycle = 5 short cycles`. Then `LW` requires 5 short cycles, `SW/ADD/SUB/JAL` require 4 short cycles, and `BEQ` requires 3. Executing `LW, LW, ADD, SUB, BEQ, SW, SW` then takes:

  - **Single-cycle**: the entire execution is rounded up to the slowest instruction, so `7 lc = 35 sc`; the ideal throughput is <u>1 instruction / 5 short cycles</u>.

  - **Multi-cycle**: the machine pays only the actual demand of each instruction, so `5sc × 2 + 4sc × 4 + 3sc × 1 = 29sc`; the ideal throughput is <u>1 instruction / average CPI</u>.

  - **Pipeline**: execution is rounded up by stage, not by whole instruction. So the total is **fill + drain** — `7 sc + 4 sc = 11 sc`; the ideal throughput (ignoring the drain tail) is <u>1 instruction / short cycle</u>, i.e. CPI ≈ 1.

- Thus, under ideal 5-stage assumptions, the throughputs are roughly **1/5, 1/(3~5), and 1**.

> ⚠️ A single-cycle CPU is limited by the slowest instruction, a multi-cycle CPU by CPI, and a pipeline by the slowest stage.

</br>

### General Formulas

(short cycle = n; number of pipeline stages = N)

- **Single-cycle**: $T = n \times T_{long}$, where the cycle length is determined by the slowest instruction.
- **Multi-cycle**: $T = \sum_i CPI(i)$.
- **Ideal N-stage pipeline**: $T = n + (N-1)$, where **drain = latency − 1**.

- **Pipeline speedup relative to single-cycle**: ideally, speedup approaches $N$ if the stages are well balanced.
- Relative to a non-pipelined multi-cycle machine, the speedup is roughly the average CPI of the original implementation.

> ⚠️ Pipelines accelerate execution by increasing **throughput**, not by reducing the latency of a single instruction. In fact, if the stages are unbalanced, individual-instruction latency may even increase.

</br>

</br>

## The Cost of Pipelining



For a pipeline to “advance one instruction every short cycle,” the following must hold:

- the hardware resources required by each stage must be able to serve different instructions simultaneously;
- otherwise, a **structural hazard** occurs: resource contention forces a stall, and the pipeline flow breaks.

For example, if IF and MEM share a single-ported memory, then whenever one `LW/SW` uses memory in MEM, the next instruction cannot fetch in IF. The fix is usually to **duplicate the resource** — separate I-cache and D-cache, or use a multi-ported memory — at increased hardware cost.

> The price of pipelining is hardware redundancy or more complicated control logic. Ideal “CPI = 1” exists only when resources are sufficient and structural hazards are resolved.

This issue is discussed further below in the WEEK 9 section.

</br>

</br>

- Clock cycle time? — 1250 ps (single-cycle), 350 ps (non-pipelined multi-cycle), and 350 ps (pipeline).
- Latency of `lw`? — 1250 ps (non-pipelined single-cycle), but **350 ps × 5 = 1750 ps** in the pipeline.

- If we move to a 6-stage pipeline, which stage should be split, and what is the new cycle time? — Split ID; the cycle time becomes 300 ps.
- Speedup? — `1250/350` relative to the single-cycle design; and an average CPI-style speedup of 4 relative to the multi-cycle design.

## Actual Implementation of a Pipeline

The rhythm of a pipeline is determined by the slowest stage. Unlike a non-pipelined multi-cycle machine, instructions in a pipeline cannot arbitrarily skip stages, because that would break the cadence.

![](https://i.postimg.cc/wMqs3dd2/jie-ping2026-03-17-14-04-32.png)

- **Sequentially**: the stages of a single instruction must still occur in order.

- **Simultaneously**: different stages of different instructions may proceed in parallel.



In particular, not every EX-stage operation finishes within one short cycle — for example, `div`. For such high-latency operations:

1. **Stall**: stop the next instruction one stage before EX, creating a bubble. This is simple, but it damages throughput and erodes the speedup of pipelining.

2. **Separate long-latency pipeline**: hand operations such as DIV / FP to dedicated units in the M-extension and F/D-extensions, allowing the main pipeline to continue advancing other independent instructions.ther independent instructions. This is one reason why [RISC-V separates such modules](https://blog.nero-lithos.com/posts/3060.2/).顺序**退休，原因是 “异常/陷阱 (traps)” 语义：如果后面的指令先退休了，而前面的指令后来触发异常，会导致程序状态不一致。然而 RISC-V **避免了很多算术异常**，所以乱序退休也是可行的。

</br>

### The true key to implementing a pipeline: insert pipeline registers

In a single-cycle machine, the control signals serve only one instruction, so there is no ambiguity. In a pipeline, however, multiple instructions simultaneously occupy different stages, which creates the following classic confusion:

- “Which instruction’s `rs1/rs2/imm` should be used right now?”
- “Which instruction’s `rd` should WB write right now?”

The solution is to **insert pipeline registers (IF/ID, ID/EX, EX/MEM, MEM/WB) between stages**, carrying both **data and control signals** forward together, so that every stage uses the information belonging to the same instruction.

![](https://i.postimg.cc/QdgpSSdr/jie-ping2026-03-17-14-47-53.png)

> Pipeline registers are **not architectural state**. Context switching saves the architectural state — registers, PC, memory-visible state — while pipeline registers, predictors, and caches belong to the microarchitectural state.

</br>

# Week 9  Pipeline Hazards



Beyond fill/drain overhead and stage imbalance, pipelines also face **data hazards**, **structural hazards**, and **control hazards**. These hazards often force younger instructions to stop in the **ID stage**, creating **stalls** and therefore **bubbles**, which reduce efficiency.

The overhead of fill and drain means that pipelines strongly prefer long-running programs —

> fill and drain are both pure overhead. Only the fully overlapped middle region is ideal. Pipelines optimize long programs; their speedup on very short code fragments may be limited.

</br>

## Structural Hazards

> A structural hazard is the most basic kind of resource conflict: the hardware is insufficient, so two instructions compete for the same resource at the same time.



#### Example A: IMEM and DMEM Port Conflict

- The IF stage needs to fetch an instruction from **IMEM**.
- The MEM stage of `load/store` instructions needs to access **DMEM**.

If the machine uses a **Von Neumann architecture**, where instructions and data share one memory port, then a data access in MEM prevents an instruction fetch in IF. That is the conflict.

![](https://i.postimg.cc/HsKMQfp9/jie-ping2026-03-18-15-08-55.png)

#### Example B: Two Adjacent Division Instructions

A divider often does not finish in one cycle. It may occupy the functional unit for dozens of cycles, such as 32 or 64. If two division instructions appear back to back, the second finds that the divider is still busy.

</br>

#### Avoiding Structural Hazards

1. **Pipeline stalls**

The most direct method is to make the younger instruction wait in ID and insert bubbles. It is simple, but hurts performance.

2. **Add resources**

In **scalar architectures**, structural hazards are relatively uncommon because designers often provision enough ports in the register file, use separate IMEM and DMEM, pipeline expensive functional units, and include multiple adders. This is the standard trade: spend hardware to buy performance.

3. **Instruction reordering**

The compiler may apply **static scheduling**, or the hardware may use **dynamic scheduling**, to separate instructions that would otherwise compete for the same resource.

4. **ISA design**

Some ISA features would require additional write ports, such as post-increment addressing. Alternatively, the ISA may simply forbid forms likely to create conflicts — for example, keeping ALU instructions away from MEM-side effects.

- For example, an R-type instruction does not truly need MEM, but letting it reach WB too early would break the rhythm of the pipeline and itself create structural problems.

</br>

</br>

## Data Hazards

> A data hazard occurs when a later instruction depends on data produced by an earlier one, but that data is not yet available. In the basic pipeline discussion, this is a **RAW** dependency.



For example:

`add x9, x0, x1`: the result is computed at the end of EXE and written back to `x9` during WB.

`sub x10, x9, x3`: it needs to read `x9` during the ID stage.

When `sub` reaches ID, `add` may not yet have written `x9` back into the register file. In the worst case, `sub` must wait until `add` completes WB, creating <u>two bubble cycles</u>.

</br>

Data hazards can be roughly divided into three categories:

### Register Access Issues

If a write and a read of the same register occur in the same cycle, the issue can be solved by **write-then-read within the same cycle**. No stall is needed.

</br>

### ALU Result Access Issues

In the diagram below, the register-access issue for `xor` is solved by write-then-read, but `sub` and `or` would still obtain the wrong value because `add` has not yet reached WB.

![](https://i.postimg.cc/j2YyTztF/jie-ping2026-03-18-18-57-58.png)

#### Solution 1: Stall the Pipeline at ID

> As in the example above, the immediately following dependent instruction may have to **stall for 2 cycles**.

This is a form of **interlocking** … which is terrible for performance.

#### Solution 2: Data Forwarding (Bypassing)

> Instead of waiting for the value to be written back into the register file, we directly **bypass** the already-computed result from the pipeline to the ALU.

This works because by the time the dependent instruction reaches EX, the producer has at least completed its own EX stage and entered MEM, so a usable copy of the result already exists inside the pipeline.

This is one of the central mechanisms by which modern pipelines solve ALU data hazards.

</br>

### Load Issues

> A `load` produces its result only after the memory access in MEM, so an immediately following dependent instruction usually **must stall for 1 cycle**.

In reality, the latency of a load depends heavily on where the data comes from: L2 may cost 2–3 cycles, L3 around 20 cycles, and DRAM roughly 200 cycles.

</br>

### Detecting Data Hazards

> Does the destination register `RegRd` of an older in-flight instruction match the source register `RegRs` that the current instruction wants to read?

Bypass from the EX/MEM pipeline register:

- EX/MEM.RegRd = ID/EX.RegRs1
- EX/MEM.RegRd = ID/EX.RegRs2

Bypass from the MEM/WB pipeline register:

- MEM/WB.RegRd = ID/EX.RegRs1
- MEM/WB.RegRd = ID/EX.RegRs2 

Additional conditions:

- If the older instruction does **not** write a register at all, then no forwarding is needed. So we must check `RegWr`.
- If the destination register is `$0`, we also do not forward, because the zero register is always 0. So we must check `RegRd ≠ 0`.

</br>

### MIPS

> **MIPS = Micro-Processor without Interlocked Pipe Stages** — the hardware tries to avoid complicated pipeline interlocks, leaving many data-hazard issues to the compiler.

This is exactly where [code scheduling](https://blog.nero-lithos.com/posts/3060p3/) enters the picture: the compiler reorders instructions to avoid **WAW, WAR, and RAW** problems; if no suitable instruction can fill the gap, it inserts `NOP`s. Even on non-MIPS machines, the same idea can reduce interlocks and therefore reduce stalls.

![](https://i.postimg.cc/CL87tspj/jie-ping2026-03-18-19-49-44.png)

</br>

</br>

## Control Hazards

> Before a branch has been resolved as **taken / not taken**, or before a jump target has been computed, the next IF stage does not yet know what instruction stream should actually be fetched.

- Control hazards arise from conditional branches and jump instructions such as `jal` and `jalr`.

- Branches typically account for **15% to 30%** of instructions — roughly one out of every 4–5 instructions. So control hazards are one of the key determinants of pipeline performance.

- ⚠️ The hardest control hazard to predict is usually the **indirect jump** form, especially `JALR`.

⚠️ The problem is <u>not the abstract existence of a jump itself</u>, but the fact that the true next fetch target is not known until the branch or jump has been resolved by the execution hardware. That is why both `jal` and `jalr` create control hazards.

Let us first discuss the taken / not-taken problem for branches.

![](https://i.postimg.cc/Bbj3swBJ/jie-ping2026-03-18-20-05-29.png)

- Without protection, as many as **three wrong-path instructions** may be issued after a branch, depending on how late the branch is resolved.

- If we simply **flush** all wrong-path instructions, and 20% of instructions are branches with 50% taken, then flushing 3 instructions creates a baseline penalty of `3 × 0.2 × 0.5 = 30%` CPI penalty.

- If the branch outcome can be resolved earlier, the flush penalty drops. That is why branch resolution should happen as early as possible.

> **The earlier the branch outcome is known, the better.**

</br>

### Strategy 1: Predict Not Taken

The simplest prediction strategy assumes that every branch is **not taken**, so the next PC is provisionally set to `PC + 4`, and the guess is checked later. In theory this cuts the penalty substantially. However, real programs are dominated by loops, where the backward branch is taken many times and not taken only once. So the practical accuracy is often only about **30–40%**.

### Strategy 2: Static Branch Prediction

> **Static prediction** means the prediction is chosen mostly at compile time and does not adapt much at runtime.

For example:

- **Backward branches** are predicted **taken**, because loop back-edges are usually taken repeatedly and fail only once at loop exit.

  For example, `beq r3, r4, -8` → predict taken.

- **Forward branches** are predicted **not taken**, because compilers often place the “then” path along fall-through order.

  For example, `beq r3, r4, 8` → predict fall-through.
  
- The prediction may be recorded explicitly in the ISA, or encoded through conventions such as operand ordering, though such schemes are limited and instruction-specific.

Even so, static prediction usually reaches only **60–70%** accuracy, or perhaps **80–85%** with stronger compiler analysis.

</br>

### Strategy 3: Dynamic Branch Prediction

> **Dynamic prediction** learns from runtime history and continually updates its guesses. Its accuracy can often reach **90%–99%**.

#### BHT: Branch History Table

A BHT is indexed using information derived from the branch PC, and stores recent branch outcomes. The table is consulted during **IF**, and the actual outcome later updates the entry. If no useful history is present, the default is often NT.

A full tag would be expensive, so real machines often index the BHT using only the low bits of the PC. This creates **aliasing**: different branches may collide into the same history entry and pollute one another. Untagged BHTs are simple and cost-effective, while higher-end processors often prefer tagged variants to reduce aliasing.

#### 1-bit Predictor

(if it was T last time, predict T this time, and vice versa)

⚠️ A 1-bit predictor suffers the classic “two mispredictions per loop” problem, especially in **nested loops**, so its accuracy is only around **70–80%**:

1. Most iterations of the inner loop are still in the looping phase, so they are T;
2. On the final loop exit, the real outcome suddenly becomes NT, causing the first misprediction and updating the predictor to NT.
3. The next time the inner loop is entered, the first iteration should actually be T, but the predictor still remembers NT, causing the second misprediction.

#### 2-bit Predictor (FSM)

> It changes direction only after two consecutive mistakes.

![](https://i.postimg.cc/1Xt0JDch/jie-ping2026-03-18-20-58-52.png)This is a **finite-state machine**, and its accuracy is often around **85–95%**.

</br>

### Strategy 4: Correlated Prediction

Also called **two-level adaptive branch prediction**, it does not ask only what this branch itself did last time, but rather:

> **whether the recent global pattern of branch behavior helps predict the current branch.**

#### 1. GHB: Global Branch History Register

The GHB is a shift register that records the outcomes of the most recent branches. Each bit encodes T/NT, so the register captures the recent control-flow context.

#### 2. PHT: Pattern History Table

The PHT stores **2-bit saturating counters**. The current GHB pattern is used as the index. If the branch turns out taken, the corresponding counter is incremented; otherwise it is decremented. So the table learns not “what this branch always does,” but “what tends to happen under this recent global history.”

For example:

```c
if (a==2) a=0;   // B1
if (b==2) b=0;   // B2
if (a!=b) {...}  // B3
```

If B1 is T and B2 is also T, then both `a` and `b` were set to 0, so B3 is likely **not taken**. The result of B3 is correlated with the results of the two preceding branches.

</br>

> Raising prediction accuracy from 98% to 99% is not “just one more percent.” It can effectively cut the **misprediction rate** in half, which matters much more for performance.

</br>

### We Have Discussed Whether to Jump — But Where Do We Jump To?

Even with good predictors, there may still be a **1–2 cycle penalty**, depending on whether the target address can be computed in ID or only in EX.

> Branch prediction has two distinct problems: **whether** the branch is taken, and **where** the target is.

To solve the “where” problem, modern processors use the **BTB (Branch Target Buffer)**, which is essentially **a cache of target addresses**.

- The target address is usually stable, because the branch immediate is stable.

- So we build a small PC-indexed cache. During IF, the machine checks the BTB using the current PC.
- If the BTB hits and the branch is predicted taken, the machine can immediately fetch the target instruction rather than waiting for later stages to compute the address.

That is why modern processors often use BHT and BTB together: the **BHT** predicts **whether**, while the **BTB** supplies **where**.

</br>

</br>

## How Do Control Signals Flow Through the Pipeline?

> **Pipeline registers, the BHT, and the BTB all belong to the microarchitecture.**

**EX stage**

The datapath needs the values of `rs1`, `rs2`, the `PC`, and the immediate.

The control signals include [Asel](https://blog.nero-lithos.com/posts/3060.4/), `Bsel`, and `ALUsel`.

**MEM stage**

The datapath must carry `ALUout` as an address, plus the value of `rs2` for stores.

The relevant control signals are `MemRW`, `BrLt`, and `BrEq`.

**WB stage**

Possible write-back sources include `ALUout`, `PC+4` (for the link register), and `MEMout`.

The control signals include **`Wbsel`, `rd`, and `WRen`**.

**Delayed use:**

- EX-stage control signals are actually used **one cycle later**.
- MEM-stage control signals are used **two cycles later**.
- WB-stage control signals are used **three cycles later**.
- The destination register `rd` must also be carried all the way to WB.

When an instruction is decoded in ID, the control unit already knows what will be needed later, but the instruction has not yet reached EX/MEM/WB. So these control bits cannot disappear; they must be stored in the **pipeline registers** and carried forward stage by stage.

</br>
