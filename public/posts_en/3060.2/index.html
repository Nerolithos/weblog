Week 3 Process of Data

Integer Arithmetic

Because the C family is historically foundational, its data types are not dynamically represented the way they are in Python. As a result, arithmetic overflow has caused a huge number of serious bugs.

⚠️ Overflow for an n-bit signed integer often appears as wraparound — $u = (a + b)\bmod 2^n \quad$.

  • Positive overflow: once the value exceeds the upper bound, subtract 2n and wrap into the negative range.
  • Negative overflow / underflow: once the value goes below the lower bound, add 2n and wrap into the positive range.

Overflow detection for two’s-complement addition:

int32_t sign(int32_t x) {
    return (uint32_t)x >> 31;
}
bool add_overFlow(int32_t a, int32_t b, int32_t r) {
    return ((~(sign(a) ^ sign(b))) & (sign(a) ^ sign(r))) != 0;
}

Modern hardware usually does not trap on ordinary integer overflow at the logic-gate level, because that would impose a large performance cost, and some domains — such as hashing and cryptography — deliberately rely on wraparound behavior. So overflow is usually handled in software.

Precise exceptions are difficult because instructions may execute out of order. Fast instructions may finish first and write their results before an earlier slow instruction raises an exception. To preserve the illusion of a sequential machine, the CPU needs significant extra machinery: internally the machine may run out of order, but externally it must still appear in order.


1-bit Full Adder, Addition, and Subtraction

Carry behavior in addition has two ingredients: propagate and generate.

For two binary numbers A and B, the i-th bit receives a carry-in Cin from bit i−1, and its sum bit is: Si = Ai ^ Bi ^ Cin

The carry-out is: Cout = (Ai & Bi) | (Cin & (Ai ^ Bi))

A concrete C++ implementation can simulate carry look-ahead (CLA) with shifts:

static uint32_t add(uint32_t a, int32_t b) {
    uint32_t p = a ^ b;
    int32_t g = a & b;
  
    g = g | (p & (g << 1));
    p = p & (p << 1); 
    // carry look-ahead: repeatedly double the carry-propagation range, <<1, <<2, <<4, ... until the full width is covered
    return (a ^ b) ^ (g << 1);
}

Subtraction is then easy, because A − B is equivalent to A + (~B + 1).


Multiplier

As in decimal multiplication, multiplying two numbers means multiplying the first number by each digit of the second number, with the appropriate place value. For unsigned binary numbers this becomes $a \times m = \sum_{i=0}^{n} m_i (a « i)$. Just as schoolbook multiplication shifts left once per digit, binary multiplication uses left shift to represent higher place value.

       0010       A  
     × 1011       B            C = 0u
-------------------------------------- "Shift and Add Method"
       0010   if(B & 1u) C += (A << 0)
      0010    if(B & 2u) C += (A << 1)
     0000     if(B & 4u) C += (A << 2)
    0010      if(B & 8u) C += (A << 3)
--------------------------------------
 = 00010110       C

For signed integers, the main additional issue is that the sign of the result is determined by A ^ B.


Divider

Schoolbook binary division works by scanning from the highest bit of the dividend downward, repeatedly checking whether the divisor can be subtracted. If it can, the corresponding quotient bit becomes 1; then the process continues.

One implementation uses a mask such as 1u << 31, shifts it right logically, extracts each bit of the dividend with AND, appends that bit into a temporary remainder register, and then tries one subtraction per step. If the result is nonnegative, the current mask bit is recorded in the quotient.



Floating-Point Numbers and Their Arithmetic

Compared with integers, floating-point numbers are inexact approximations of real values. Recall the IEEE single- and double-precision formats:

type sizeof(～) composition composition size
float 4 Byte sign, exponent + 127, fraction 1b + 8b + 23b
double 8 Byte sign, exponent + 1023, fraction 1b + 11b + 52b

⚠️ Bias: to represent negative exponents, we store the exponent after adding an offset.

For single precision, the exponent field leads to several cases:

  • If e is neither all zeros nor all ones (normalized), then $$x = (-1)^s \times (1+\frac{f}{2^{23}})\times 2^{(e-127)}\ \ (s:sign, f:fraction, e:exponent)$$
  • If e = 00000000 (denormalized / subnormal), then when f = 0 the value is zero; otherwise $$x = (-1)^s \times \left(\frac{f}{2^{23}}\right)\times 2^{-126}$$
  • If e = 11111111, then $$F=0: \pm\infty,\ \ F ≠ 0: NaN$$

⚠️ Note: in the denormalized case, the effective exponent is fixed at −126; the usual hidden leading 1 is absent. For example, 0 00000000 1000... equals 2-127.


What is a floating-point number, really?

⚠️ The key idea is that range and precision cannot both be maximized. Floating point allocates more precision near zero and less precision for huge magnitudes. Concretely, normalized single-precision values in the interval 2e to 2e+1 are spaced by 2e-23, while denormalized values have fixed spacing 2-149.

The point of the denormalized region is to create a dense band of representable numbers between 2-149 and 2-126. This gives us gradual underflow: tiny positive values approach zero smoothly under rounding instead of collapsing abruptly to 0.

So floating point sacrifices precision for very large numbers in exchange for a vastly larger representable range, with finer precision concentrated near zero. That is why the “point” effectively floats.


Binary Expansion

To convert the fractional part of a decimal number into binary, repeatedly multiply by 2 and record whether the result crosses 1:

  • Standard binary-expansion method: (1) separate the integer and fractional parts; (2) repeatedly multiply the fractional part by 2 and take mod 1 until a pattern emerges.

Rounding

IEEE defines four rounding modes: round-to-even, round-toward-zero, round-down, and round-up. The default is usually round-to-even. Converting a float to an integer typically uses round-toward-zero. In single precision, integers larger than 224 are no longer all exactly representable.


Floating-Point Arithmetic

  • Addition and multiplication remain commutative, but associativity fails: (2 + 2^100) - 2^100 ≠ 2 + (2^100 - 2^100).
  • Converting int32_t to float is exact only within the representable range, while converting float back to int32_t may involve rounding or even overflow.
  • Floating-point addition typically proceeds in stages: align exponents, add significands, normalize, then round and renormalize if necessary.



Week 4 Machine-Level Programming

Understanding Machine-Level Code Through Assembly

  • x86 is a large CPU ISA family descended from the Intel 8086. We usually study x86-64 (AMD64), the 64-bit extension of x86. In 32-bit mode the registers are %eax, %ebx, etc.; in 64-bit mode they become %rax, %rbx, and so on. The program counter is %rip. x86 is fundamentally a 2-operand machine.
  • ISA (instruction set architecture) is the CPU specification as seen by software. Different hardware ecosystems define different ISAs — for example CISC (x86), where hardware is more complex, versus RISC (ARM, RISC-V), where hardware is simpler but software carries more responsibility.
  • What the machine actually executes is a sequence of bytes — hexadecimal byte strings, each encoding one instruction. The machine knows almost nothing about source-level constructs such as variable names or data types.
  • Assembly language is the human-readable form of machine code under a specific ISA. Two common ways to inspect machine code are:
    1. Force the compiler to emit assembly directly: gcc -Og -S xxx.c
    2. Use a disassembler to translate byte sequences into something resembling assembly: objdump -d xxx.o (Linux)
  • Disassembly output usually contains the address / offset, the raw bytes, and the equivalent assembly instruction.
  • A major feature of x86-64 is that instruction length is variable (1–15 bytes). More common instructions or those with fewer operands are often shorter.
  • Disassembly works by reconstructing instructions from bytes and does not require source code. Its naming conventions may differ slightly from GCC-generated assembly.


x86-64 (optional)

Accessing Information — Registers

A 64-bit CPU has one program counter register (%rip) and 16 general-purpose registers (%rax, %rbx, …). These registers are flexible in purpose, except that %rsp is conventionally fixed as the stack pointer.

  • All 16 of them are 64-bit integer registers used to store integers and pointers.
  • In general, the register width matches the machine width. Suffixes such as b, w, l, and q control whether one accesses 8, 16, 32, or 64 bits.

Common x86-64 Assembly Instructions

  1. Data movement / address calculation
  • mov a, b: copy data from a to b. Depending on the operand width, we have movb = 8b, movw = 16b, movl = 32b, and movq = 64b. The same suffix idea also appears in commands such as call, ret, and pop, though the suffix is often omitted in practice.
  • lea: load effective address / perform address arithmetic. It is often used for pointer arithmetic or multiply-add style computations. Unlike mov, it does not read memory.
  1. Stack operations
  • pushq a: expand the stack toward a lower virtual address by 8 bytes, i.e. %rsp -= 8, and then push a onto the stack.
  • popq a: pop the top stack value into a (register or memory), i.e. %rsp += 8 afterward.
  1. Arithmetic
  • add / sub / neg: addition / subtraction / negation.
  • imul: multiply (in several forms); idiv: divide (typically with rdx:rax as the dividend).
  • inc / dec: increment / decrement (often replaced by add/sub $1).
  • and / or / xor / not: bitwise and / or / xor / not.
  • shl / shr: logical left / right shift; sar: arithmetic right shift for signed values.
  1. Function calls
  • call a: push the return address (the next instruction) onto the stack, then jump to the target.
  • ret: pop the return address from the stack and jump back.

x86-64 Operands and Addressing

Three major operand classes:

  • Immediate: a constant, written with $ in AT&T syntax (for example $1, $0x42).
  • Register: such as %rax, %rbx, and so on.
  • Memory reference: compute an effective address through an address expression, then access memory at that location.

Memory-addressing form: Imm(r_b, r_i, s)

  • The effective address is built from four parts: displacement Imm, base register r_b, index register r_i, and scale factor s.
  • It is computed as M[Imm + R[r_b] + R[r_i]·s], where R[...] denotes the contents of a register and M[...] denotes the contents of memory at that address.
  • The scale factor s must be 1/2/4/8, matching common basic-type sizes and making expressions such as “array index × element size” convenient.
  • Examples: $0x42 denotes the literal constant 0x42; 0x42 denotes the data stored at memory address 0x42; %rax denotes the value stored in that register; and (%rax) denotes the data at the memory location pointed to by the value in %rax.


RISC-V

The x86 ISA is bizarre enough that it feels like someone wandered into a CS classroom while being neither an Arknights player nor a femboy. Let us return to the comfort of RISC.

RV32I (RISC-V 32I) is the minimal mandatory 32-bit integer instruction set in the RISC-V family. It follows a load/store architecture and contains 32 general-purpose 32-bit registers (x0–x31, with x0 hardwired to zero), one 32-bit PC, and several CSR registers. It is the base on which all RISC-V processors are built.

Key features and components of RV32I:

  • The “I” in RV32I means the base integer ISA. There is also E, a reduced embedded profile, plus many extensions: M for multiplication/division, A for atomic memory operations, C for compressed 16-bit instructions, F/D for floating point, V for vectors, and so on.

  • Running an instruction from an unsupported extension — for example, mul on a pure RV32I machine — triggers an illegal instruction exception.

  • FMA is especially important in the floating-point extensions. Unlike the usual 3-operand RISC style, it is effectively a 4-operand operation, which is highly useful in matrix and vector arithmetic.

  • Instruction classes: RV32I uses six encoding formats — R, I, S, B, U, and J.

  • RISC is generally 3-operand: the pattern is A = B op C, as in ADD x6, x0, x7.

  • Load/store architecture: memory is accessed only by load and store instructions; all other computation takes place between registers.

  • Multiplication/division and floating-point arithmetic are not part of the minimal base; they belong to separate extensions.

  • x0 is always zero, because zero is used constantly. For example, negation can be written as sub x9, x0, x8.

  • More registers are not automatically better. A larger register file can help performance, but it also lengthens instructions and can slow register access.


RV32I Instruction Categories

By functionality:

  • ALU instructions (register-register, register-immediate)
  • Control-flow instructions (branches and jumps)
  • Memory access instructions (load and store)
  • CSR instructions (related to privilege and exceptions, often read-modify-write)
  • Privileged instructions (kernel / OS only)

⚠️ By encoding format:

In RV32I, each instruction is 32 bits wide, and its fields are arranged according to one of six fixed formats (RISBUJ).

  • opcode is a 7-bit operation code telling the CPU which major instruction class this belongs to.
  • rd means register destination, specifying which target register receives the result.
  • rs means register source, specifying which register is read.
  • funct3 / funct7 further refine the operation, such as distinguishing add from sub or srl from sra.
  • imm is the immediate field, used as a literal value, an offset, or a jump distance depending on context.

⚠️ The six major instruction-format families are R / I / B / J (the common ones), plus S for stores and U for upper-immediate instructions:

  • R: no immediate field; used for many register-register operations (opcode 0110011), such as add, sub, and, and sll.
  • I: 12-bit immediate; used for register-immediate operations (0010011) such as addi, andi, and also ⚠️ all load instructions (0000011) such as lw, lb, lh.
  • B: 12-bit immediate; branch instructions with no rd, such as beq (branch if equal) and bne (branch if not equal).
  • J: 20-bit immediate, for instructions such as jal (jump and link).
  • S: store-to-memory instructions with no rd, such as sw, sb, and sh.

ALU Instructions

R-type: compute using two register operands and write the result back to a register — OP: rd ← rs1 (funct3, funct7) rs2.

  • For example, a sub encoding can negate the value in x6 and store it back into x6.

  • Likewise, an add encoding with x0 can effectively copy the value in x6 into x7.

I-type: compute using one register and one 12-bit immediate, then write the result back — OP-IMM: rd ← rs1 (funct3) I-imm.

  • For example, an addi instruction may add 1024 to x6 and store the result in x7.
  • Shift-immediate instructions are also I-type, but here the immediate is not treated as an ordinary 12-bit signed value. Its low bits form the shift amount (shamt) (5 bits in RV32I, 6 bits in RV64I), while higher bits — including inst[30] — distinguish SRLI (0) from SRAI (1).
  • For example, “bitwise NOT of x1” can be written as xori x1, x1, -1.

Load Instructions

I-type: use a base register plus an immediate offset to form a memory address, then load from memory into a register — LOAD: rd ← memory[rs1 + I-imm].

  • For example, lw may fetch 4 bytes from address [x6] - 7 into x7. A 12-bit signed immediate covers offsets in the range [-2048, +2047]. Such offsets are often used to access array elements or fields inside container-like structures.
  • More concretely, suppose we have an image-pixel array Struct Color {int r, int g, int b} pixelArray[]. If the base address is stored in x1, then the green component of the 100th element could be accessed with an instruction such as lb x2, 1204(x1).
  • lb reads 1 byte, lh reads 2 bytes, and RV64I also provides ld for 8-byte loads.

Store Instructions

S-type: store the contents of a register into memory at an address formed by register + immediate — STORE: mem[rs1 + S-imm] ← rs2.

  • For example, sw may store the full word in x6 to memory at address [x7] + 1. The immediate field in the S-type encoding is comparatively small and is split across the instruction format.

⚠️ Important pitfall — data alignment: when storing n bytes, the address should normally be a multiple of n; for example, sw / lw usually require addresses divisible by 4. RISC-V itself is not absolutely strict in every implementation, but compilers generally enforce alignment, and many RISC designs such as ARM assume aligned access. Alignment helps avoid crossing page, cache-line, or other storage-unit boundaries.

Example:

  • If the fields require alignments 4, 8, 1, 4, 4, then padding is needed: 4 bytes before count, 3 bytes before red, and the overall structure alignment equals the maximum field alignment = 8 bytes. Thus the full structure occupies 32 bytes, and the offset of the 100th red field is 32×100 + 4 + 4 + 8 + 1 + 3 = 3220.

Branch Instructions

SB-type: compare two register values and choose one control-flow direction, for example with beq.

  • In S-type, the offset is relative to a base address and measured in bytes. In SB-type, the offset is relative to the PC and measured in halfwords (2 bytes). Taken together, the encoding lets a branch reach roughly ±4 KB of instruction space.

  • For unconditional jumps, one uses J-type jal, which can reach about ±1 MB, or I-type jalr, which can cover the full 4 GB address space in RV32 through register-based addressing.

  • RV32I includes >= and < forms such as bge and blt, but not separate > or <= mnemonics. It also provides unsigned forms such as bgeu and bltu. A neat trick is that a range test like 0 <= x < y can often be done with a single unsigned comparison bgeu x, y, which simultaneously excludes negative values and values above the upper bound.

Thus, if the branch is taken, the machine updates PC += imm × 4; otherwise the default next step is simply PC += 4.


LUI and AUIPC

U-type: used to construct larger constants or addresses that exceed the small immediate range (for example, beyond ±1MB).

LUI: place the 20-bit upper immediate into a register, then shift it left by 12 and fill the low bits with zeros: rd = U-imm << 12.

AUIPC: build a large PC-relative address base: rd = PC + (U-imm << 12).

  • For example, to build a 32-bit constant such as 0x12345678, one may use lui x1, 0x12345 and then addi x1, x1, 0x678.

    ⚠️ Important pitfall: when the low 12 bits of the desired 32-bit constant are ≥ 0x800, the naive LUI + ADDI combination may fail because the ADDI immediate is signed and will be sign-extended. In that case the decomposition must be adjusted.

  • Alternatively, one may use the pseudo-instruction li and let the assembler handle the details.


Jump Instructions

A jump is, conceptually, much like a branch without a condition.

J-type here refers specifically to the unconditional jump-and-link instruction jal — JAL: rd ← pc + 4; pc ← pc + J-Imm.

  • One might wonder why rd is not repurposed to enlarge the immediate field. The reason is that doing so would introduce an additional instruction format and complicate decoding, which goes against the “RISC” design spirit. For longer-range control flow, JALR can be used instead.

I-type here specifically includes the unconditional indirect jump jalr — JALR: rd ← pc + 4; pc ← (rs1 + I-Imm) & ~0x01.

  • ⚠️ For a return, one often sees a pattern like jalr x0, 0(x1). If the JALR base register is x1 and the offset is 0 or very small, it is often reasonable to guess that the instruction is a return — though this guess is not infallible, because x1 is not magically reserved by hardware for returns.

  • In both cases, the lowest bit of the jump target is forced to 0, ensuring alignment compatible with 16-bit instruction support.

  • Conceptually, JALR is often used together with address-construction instructions such as AUIPC, which lets it indirectly jump anywhere in the full 32-bit virtual address space (0x00000000 to 0xFFFFFFFF).


CSR Instructions

These are system-level instructions that use special CSR (control and status register) registers to manage program execution — exceptions, interrupts, performance counters, timers, and so on.


Exception Handling

Two things the hardware must do:

  1. Record the exception, including key CSR state such as the following:

  • mepc – the PC of the instruction that raised the exception

  • mcause – the cause of the exception

  • mstatus.mpp – the privilege mode active at the time of the exception, so the system can later restore it

  1. Hand control to software: the CPU reads mtvec (the trap-handler address), sets the PC there, and switches to a higher privilege level.

Software Side (Exception Handler)

A common design is one Common Handler (CH) plus several Interrupt / Exception Handlers (IH).

CH (common entry) is usually written in assembly and performs the “dirty work.”

  1. Save all GPRs (general-purpose registers)
  2. Prepare arguments and call a C-level IH function
  3. After the IH returns, decide how mepc should be updated (+4 or +2)
  4. Restore all GPRs
  5. Execute ERET (usually called mret in RISC-V)

IH (specific handlers) are often written in C and dispatch according to mcause.

  • illegal instruction → illegal_inst_ih

  • timer interrupt → timer_ih

  • external interrupt → ext_ih

  • ……



RV32I ABI

ABI = Application Binary Interface. It specifies how separately compiled binary programs cooperate with one another.

If different assemblers and toolchains used different register conventions, chaos would result. So, taking the RISC-V ABI (ILP32) as an example:

At the hardware level, registers usually have few absolute restrictions, but the ABI assigns them conventional roles.

  • Caller-saved: t0–t6, a0–a7, and ra. This means that if the caller wants to preserve those register values across a function call, the caller must save them first. The callee is free to overwrite them. They are commonly used for temporaries and argument passing, and a0/a1 also often hold return values.

  • Callee-saved: s0–s11 (with sp treated specially and maintained carefully). If the callee uses these registers, it must first save the old values and restore them before returning. This allows the caller to keep “long-lived variables” in the s registers across function calls.
