January 6, 2026

As one great philosopher once said: “There are three kinds of people in computer science: Turing, von Neumann, and Arknights / VTuber addicts.”


Week 1 Introduction

Computer System = Hardware + System Software

We want computers to …

  1. do more (more transistors, more cores …)
  2. run faster (concurrency, optimization, cache …)

This course leans more toward the software side: how code is compiled, how code is optimized, and related topics. The textbook is Computer Systems: A Programmer’s Perspective (3rd ed.).


Why Binary?

Why did von Neumann choose binary?

From a hardware perspective, modern CMOS technology roughly follows the dynamic-power relation P ≈ k·C·U2·f (load capacitance × voltage squared × switching frequency). Circuits operating far above about 1 V would face exponentially worse leakage, breakdown, and burnout risks. As fabrication keeps shrinking, high-voltage designs become unrealistic both in stability and cooling requirements. So we use electrical signals around 0–1 V to represent information.

Under this constraint, voltage error is common, so a sufficient noise margin is necessary. For instance, we cannot define “0” and “1” as 0.1 V and 0.2 V, because ordinary electrical noise would make them hard to distinguish in practice. Binary is therefore the most stable choice: values near 0 V mean “0,” values near 1 V mean “1,” and noise is unlikely to make the two collapse into one another.

Beyond that, vacuum tubes, relays, and transistors are naturally switch-like, bistable devices. Much of the physical world is easier to model in bistable terms, and even human logic itself — Boolean algebra — is built around binary distinctions. It is hard to imagine a more concise logical system than yes/no.

At the deepest level, one might joke that “the universe wants to destroy computers,” because stability and determinism run against entropy. Binary is simply the most deterministic and least ambiguous representation we have.



Hardware: CPU

  • The CPU (processor) consists of one or more minimal processing units called cores. A core can independently execute at least one thread of control, so we distinguish between “multi-core parallelism and single-core concurrency,” which we will discuss further later.

  • A CPU does not operate as one continuous flow; instead it repeatedly follows a “fetch, decode, execute” cycle: fetch an instruction from memory/cache, decode it, then compute or access memory. The smallest time unit is a cycle, and frequency is measured in GHz. For example, a 3 GHz CPU executes 3×109 cycles per second.

  • The CPU consists of the following basic parts:

    PC (program counter) is part of the CPU’s architectural state. It is a special instruction register that stores a pointer to the current instruction in main memory. The CPU processes the instruction pointed to by the PC and then updates the PC to the next one.

    Registers: the smallest and fastest storage structures in the machine, each storing one word (that is, an n-bit value on an n-bit machine).

    Register file: a temporary data-storage component consisting of a collection of general-purpose registers, usually only a few hundred bytes in total, directly connected to the ALU.

    ALU: the arithmetic and logic unit, which operates on one or more values from the register file.

    SRAM L1–L3 caches: the high-speed cache hierarchy connected to the register file.

    bus interface: the CPU’s communication interface to the system bus.



Hardware: Bus

  • CPUs, memory, and I/O devices communicate through physical wires called the bus, which transfer word-sized chunks of bytes. The common signal classes are data lines, address lines, and control lines. Slow memory access, and even slower I/O, easily create bus congestion. This is one reason why technologies such as caches and DMA were developed.

  • Direct Memory Access (DMA) moves data directly between a device and memory, but it does not execute instructions. It can bypass some CPU-internal movement, but it still depends on the I/O bus and memory bus.



Hardware: Memory Hierarchy

Memory Hierarchy

Storage Type Access Time Location
L0 Registers ~1 cycle CPU
L1 Cache ~4 cycles CPU
L2 Cache ~10 cycles CPU
L4 DRAM main memory ~100–300 cycles beyond the I/O bridge
L5 Local SSD (e.g. disk) ~105 cycles beyond the I/O bridge
L6 Web Server \ outside the computer

The hierarchy means that an upper layer can serve as a cache for the layer below it. For example, DRAM acts as a cache for SSD-backed storage. Moving upward, capacity decreases and cost per byte rises, but speed increases.

This hierarchy exists because main memory is slow, because programs exhibit the two kinds of locality, and because processes must be isolated from one another.

The point of multiple cache levels is to reduce timing uncertainty in access by expanding the amount of recently useful data that can be retained.


Main Memory (DRAM)

Main memory sits outside the CPU and is a temporary storage layer. Files usually reside on disk when inactive, and only the working subset is cached into DRAM while the program is running.

Why is main memory so slow? DRAM = Dynamic RAM: it uses the “charged / uncharged” state of capacitors to represent bits. Because capacitors leak, DRAM must be periodically refreshed. As a result, main memory is slow, cheap, and large.

That is why we need CPU caches to bridge the time gap between the register file and main memory.


CPU Cache (SRAM)

CPU cache is a small but fast storage layer located between the CPU and DRAM. SRAM stores bits using the bistable state of flip-flop circuits. Caches are fast, expensive, and small.

  • Caching works by placing the data most likely to be used soon close to the CPU, so that slow DRAM does not choke the processor’s much faster execution speed. The basis for what is “most likely” is temporal and spatial locality:
  1. Temporal locality: after accessing A, the program is likely to access A again soon.
  2. Spatial locality: after accessing A, nearby locations such as A−1 or A+1 are likely to be accessed as well.


Program Lifetime

  1. Source code (ASCII text):
printf("hello, world\n");]
  1. Compilation:
gcc -o hello hello.c

This is typically described in four steps: preprocessor: macro expansion, processing include and define, removing comments, and turning .c → .i

compiler: semantic checking, then translating ASCII .i into assembly .s

assembler：assembly(.s)→binary(.o)

linker：pack object files and libraries into executable, .o→.exe

  • In practice, the compiler may skip an explicit assembly output step and go directly toward binary unless the -S flag is used.
  1. Produce the executable file (binary instructions) and store it on disk.
  2. Execution:
./hello

A more concrete picture is the following:

  • After ./hello is entered, the shell asks the OS to handle the command. The command flows through the I/O subsystem into the CPU, the executable is DMA-transferred from disk into DRAM page by page, then fetched upward through the cache hierarchy into registers, and finally the CPU instructs the display subsystem to show the result.

  • The shell is the software layer that connects user input to the OS. Different systems provide different shells.
  • The OS spends much of its time moving data, not merely performing arithmetic.
  • The time gap between main memory and registers is what makes caching necessary; cache hits are central to performance optimization.


Software: Operating System

The OS is the software layer that schedules and manages hardware resources. It abstracts hardware into manageable virtual objects:

  1. Process: an abstraction over CPU, memory, and I/O resources. A process may contain multiple threads.

    • Thread: a sub-execution path inside a process that shares the same virtual memory. Threads communicate quickly, but can also interfere with one another.

    • Concurrency: multiple processes conceptually coexist at the same time, for example by alternating execution. On a single-core processor, the OS kernel implements concurrency through context switching, letting different processes take turns on the CPU while maintaining the illusion that each one owns it.

    • Parallelism: multiple processes really run at the same time. It is a special case of concurrency. Multi-core or multiprocessor systems can execute different processes simultaneously, rather than merely switching between them.

    • Note: what follows concerns non-process-level virtual parallelism, which is a completely different concept.

      ILP (instruction-level parallelism): “doing several things at once with the same body.” It does not require multiple cores; for example, different execution units may perform addition and multiplication in parallel.

      DLP (data-level parallelism): “doing many similar things at once with carbon paper.” It also does not require multiple cores; SIMD is the standard example.

    • Hyperthreading: one physical core is exposed as multiple logical CPUs, helping the machine schedule concurrent or parallel work more effectively.

  2. Virtual Memory: an abstraction over main memory plus disk-backed storage; the virtual address space is ultimately backed by disk.

    • The partitioning of virtual memory is a semantic and safety model. It does not correspond directly to hardware structures such as registers, caches, or DRAM.

  • The heap grows upward and the stack grows downward, leaving unallocated virtual space in between. Virtual memory is managed by pages, and page 0 is left unmapped so that null-pointer errors can be caught.
  • Virtual memory is usually byte-addressed, meaning the machine addresses 8-bit units, though words and double-words are common derived units. An “n-bit computer” has a theoretical virtual address space of 2n bytes.
  • In the hierarchy, L4 DRAM acts as a cache for L5 disk-backed virtual memory, holding only the pages currently in use.
  • The virtual address space can be much larger than physical memory. A 64-bit computer absolutely does not contain 264 bytes of DRAM.
  • Different processes may also share data: distinct virtual addresses in different processes can map to the same physical DRAM page.
  1. File: an abstraction that treats I/O resources as continuous byte streams.


Amdahl’s Law

If a fraction α of a system becomes k times faster, the total speedup is still heavily constrained by the remaining portion: $$T_{new} = (1-\alpha)T_{old} + \frac{\alpha T_{old}}{k},\ \ \text{Speedup} = \frac{1}{(1-\alpha) + \alpha/k}$$

  • Therefore, optimization should first target the slowest part of the system, not the part that is already fast.

    To substantially speed up the whole system, one must improve a substantial fraction of the total workload:

$$S_{k\to \infty} = \frac{1}{1- \alpha}$$

  • Moore’s Law does not directly translate transistor count into real system performance, because more physical resources do not automatically imply proportionally better efficiency.



Week 2 Storage of Data

For a byte-addressed machine, all data is ultimately just a sequence of bytes; data types exist mainly for the compiler. Byte addressing means each address refers to 8 bits. Thus a 64-bit machine can be abstracted as having a 264-byte virtual address space (16 EB), and every file can be viewed as some byte block inside it. (For a 32-bit machine, the limit is 4 GB.)


Number Bases

Becoming fluent in hexadecimal is essential for reading assembly and memory dumps…

  • One hexadecimal digit equals 4 bits.

  • 1111 is decimal by default, whereas 0x1111 explicitly denotes hexadecimal.

  • A bit vector writes a binary number as a vector of bits, usually read from right to left in significance order: 42 → 11010BIN → [1, 1, 0, 1, 0].

  • One byte ranges from 0000 0000BIN to 1111 1111BIN, equivalently from 0DEC to 255DEC, or from 0x00 to 0xff.

  • The hexadecimal letters a–f are case-insensitive and correspond to the four-bit patterns 1010–1111.

Base conversion:

  • HEX → BIN: expand each hexadecimal digit into a four-bit binary pattern.
  • BIN → HEX: group bits in chunks of four and rewrite each chunk as one hex digit.
  • HEX → DEC: multiply each digit by its corresponding 16n place value and add them up.
  • DEC → HEX: repeatedly divide by 16 using Euclid’s method, and read the remainders from right to left.


Integer Representation

  • In C, an integer literal such as 0x42 is signed by default unless written as 0x42u or 0x42U.
  • Integer data types (int, long, etc.) are stored directly as binary strings. For example, 12345 is represented in memory (big-endian view) as 00 00 30 39.
  • Integers come in two major forms: unsigned and signed. For an n-byte integer type, the former stores 0 to 2^(8n)-1, while the latter uses the highest bit for negative values and stores -2^(8n-1) to 2^(8n-1)-1; in the signed interpretation, a highest bit of 1 indicates a negative number.

Unsigned Integer

B2U_ω() is the bijection that maps a binary string to an ω-bit unsigned integer:

$$B2U_ω(\vec x) = \sum_{i=0}^{ω-1}x_i·2^ω $$

  • For example, a 1-byte unsigned char can represent 0–255; for an n-byte unsigned integer, the maximum value is 0xF...F (with n/4 hex digits F).

  • Once an integer calculation exceeds the available bit width, the extra high bits are discarded; this is called overflow.


Signed Integers (Two’s Complement)

  • A 1-byte signed char represents -128 to +127. In C, negative numbers are usually stored in two’s complement, meaning that the negative of a number is formed by bitwise inversion plus one. For example, if 1 is 0000 0001, then -1 is 1111 1111; if 127 is 0111 1111, then -127 is 1000 0001, and -128 is 1000 0000.
  • In hand calculation, hexadecimal is convenient: 0x3039 → invert → 0xCFC6 → add 1 → 0xCFC7.
  • ⚠️ Why two’s complement is useful: subtraction can be treated as “addition of a negative,” and the natural bit overflow still yields the correct result. For example, with char, 3 - 2 = 3 + (-2) → 0011 + 1110 = 10001 → 1.

B2T_ω() is the bijection that maps a binary string to an ω-bit signed (two’s-complement) integer:

$$B2T_ω(\vec x) = -x_{ω-1}2^{ω-1}+ {\sum_{i=0}^{ω-2}x_i·2^ω}$$

  • For an n-byte two’s-complement integer, the minimum value is 0x80...0, while the maximum is 0x7F...F.

  • Other negative-number representations also exist, such as one’s complement and signed magnitude, but none of them have the same computational advantages as two’s complement.


A classic pitfall: type casting

  • Sign extension: if a value is widened (e.g. char → int) without changing signedness, one copies the highest bit into the new high bits. For example, char: 10000101 becomes int: 1111...10000101.
  • Truncation: when a value is narrowed to a smaller type width, the excess high bits are simply cut off. This can lead to undefined behavior in some contexts.

In C, mixing signed and unsigned values is a common and dangerous logic error, because the two use different binary-to-decimal interpretations.

  • A type cast changes the interpretation of data, not necessarily the underlying bit pattern. For example, if int a = -4242; auto b = (unsigned)a;, both a and b share the same bit pattern ~(0x00001092)+1 = 0xFFFFEF6E, but b is interpreted as the unsigned integer 4294963054. In other words, a negative two’s-complement bit pattern becomes a large positive number under unsigned interpretation.

  • ⚠️ Extremely dangerous case: when one operation mixes signed and unsigned values, C implicitly converts the signed operand to unsigned before performing the calculation.


A spectacular bug …

if (int i < vec.size())   // 若 i 是负数，因为 size() 是 size_t，强转换数值起飞
for (int i = n-1; i >= 0; i--)   // 若 n 或 i 是 unsigned，都死循环
unsigned len = a - b;   // a < b 时数值爆炸

⚠️ Use unsigned for indices / subscripts, signed for loop iterators, avoid mixing signedness, and cast explicitly before comparison so that you remind yourself what is happening.



Floating-Point Representation

type sizeof(～) composition composition size
float 4 Byte sign, exponent + 127, fraction 1b + 8b + 23b
double 8 Byte sign, exponent + 1023, fraction 1b + 11b + 52b

For example, (float)12345.0 can be written as 11000000111001BIN, that is, 1.1000000111001 × 2^13, so its fields decompose as follows:

0 (代表正数) ｜13+“偏移量 (bias)” = 140 → 10001100 | “小数位” 10000001110010000000000

Therefore, the single-precision floating-point value 12345.0 is stored in memory (big-endian view) as 46 40 E4 00.

  • Bias: since exponents may be positive or negative, an exponent field is stored by adding a bias such as 2^(n-1)-1, thereby shifting it into the nonnegative range.

Strings

In C, a string is simply an array of characters represented through an ASCII-style encoding table.

For example, the string “AaZz” appears in memory as 41 61 5A 7A 00, because it is equivalent to the character array char[A, a, Z, z, \0]. Note that I do not label this with endianness, because string literals do not carry byte-significance structure in the same way numeric multi-byte data does.

Code

On different machines, the binary instructions produced from the same source file may be completely different. From the computer’s point of view, a program is just a sequence of bytes. Without source-level information, binary code is difficult to move across machines with different OSes, ISAs, and calling conventions; such binaries are said to be ABI-incompatible.



Word Size and Data Size

  • On an n-bit machine, all pointers occupy n bits. Many programmers once assumed that an int could store a pointer. That often works on many 32-bit machines, but fails on 64-bit systems.
  • A program compiled with gcc -m32 hello.c may run on both 32-bit and 64-bit environments, whereas using -m64 generally requires a 64-bit environment. This kind of backward compatibility matters for long-term file and software portability.
  • In the LP64 versus ILP64 debate during the shift from 32 bits to 64 bits, ILP64 supporters argued that int should also become 8 bytes, because many programmers had historically mixed integers and pointers on 32-bit systems where they shared the same width. LP64 supporters argued that this would waste memory and reduce performance. In practice, LP64 became the dominant choice.


Addressing and Byte Order

In C, a pointer points to the first byte of the referenced data object. Naturally, an n-bit machine uses n-bit pointers.

int val = 4242;
auto* ptr = &val;

Suppose &val is allocated at 0x0010. Then val occupies virtual addresses 0x0010 through 0x0013, and ptr stores 0x0010. For a data type that spans multiple bytes, such as 4242DEC = 0x0000109A, the value is split into separate byte chunks, and there are two possible byte orders:

Byte order in storage 0x0010 0x0011 0x0012 0x0013 Common in
Big-endian 00 00 10 9A TCP/IP, IBM, Motorola
Little-endian 9A 10 00 00 Intel (x86), iOS, Android

Big-endian versus little-endian refers to whether bytes are arranged from high significance to low significance or the reverse. That is why literals such as strings are not usually described in terms of endianness in the same way as multi-byte numeric data.

Many modern processors are bi-endian in hardware, but the operating system fixes the active byte order. For example, ARM chips in phones typically run in little-endian mode under iOS or Android.

Compilers usually hide byte order from ordinary programmers, but endianness must still be considered in the following cases:

  1. Network communication: big-endian and little-endian machines cannot blindly exchange raw binary words; communication protocols usually standardize on network byte order = big-endian.

  2. Assembly and disassembly: when reading disassembled bytes such as 4004d3: 01 05 43 0b 20 00 add %eax,0x200b43(%rip), the little-endian byte order can look counterintuitive, because the apparent byte sequence corresponds to a value like 0x200b43.

  3. Type casting: the following example shows the special power of C pointers. By type-casting an address to unsigned char*, one can read memory byte by byte, regardless of the original data type, and treat it as a “byte array.”

#include <stdio.h>
typedef unsigned char *bytePtr;
void showBytes(bytePtr start, size_t len) {
	size_t i;
	for (i = 0; i < len; i++) printf("%.2x", start[i]);
	printf("\n");
}
void showInt(int x) show_bytes((bytePtr)&x, sizeof(int)); // typecast
void showFloat(float x) show_bytes((bytePtr)&x, sizeof(float);

Running such code with x = 4242 = 0x3039, one gets the counterintuitive 39 30 00 00 on a little-endian machine, but 00 00 30 39 on a big-endian machine.



Boolean Algebra

If this topic is unfamiliar, see CSC3001 Boolean Algebra Basics.

At the hardware level, a computer fundamentally only needs one logically complete gate family, such as NAND (⊼). Because {NAND} is functionally complete, all other Boolean operators can be built from it.

~A A | B A & B A ^ B
A ⊼ A (~A) ⊼ (~B) (A ⊼ B) ⊼ (A ⊼ B) (A ⊼ (A ⊼ B)) ⊼ (B ⊼ (A ⊼ B))
  • a & (b|c) = (a&b)|(a&c)；a|(b&c) = (a|b) & (a|c)
  • a ^ a=0；(a ^ b) ^ a=b

Example: swapping data without a temporary variable:

void swap(int* x, int* y) {
	*y = *x ^ *y;
  *x = *x ^ *y;
  *y = *x ^ *y;
}

Bitwise Operations

⚠️ Three practical rules for bitwise programming:

  1. Avoid floating-point numbers entirely (for example, do not use pow()).
  2. Use fixed-width integers such as uint32_t or uint64_t.
  3. Always keep track of the exact type width, and beware of undefined behavior.

In CSC3001, Boolean algebra may look like pure logical or numerical manipulation. In CSE, however, it often appears as bitwise computation. We frequently write a subset of a finite universe as a bit vector, then perform Boolean operations on those bits. For example, a character set over the 128-element ASCII universe can be represented by 128 bits, and over the universe {1,2,3,4,5}, the subset {2,1,5} can be represented as 10011.


Shift Operations

  1. Left shift by n (<< n) moves the data n binary positions to the left, discarding overflow on the high side and filling empty positions on the right with zeros.

    For example, an n-bit network mask can be generated by uint32_t mask = 0xFFFFFFFFu << (32 - n);, such as the 24-bit mask 0xFFFFFF00.

    int x = 1 << 31; is invalid, because the 1 has been shifted into the sign bit. This is one reason why bitwise operations are safest with unsigned types.

  2. Right shift by n (>> n) divides into arithmetic right shift (replicate the sign bit on the left) and logical right shift (fill the left side with zeros).

    Most compilers use arithmetic right shift for signed integers, but for unsigned integers the shift must be logical. Otherwise the meaning of the bit pattern changes incorrectly.

  • Both left and right shifts associate to the left, and their precedence is lower than ordinary arithmetic. Thus 1+2 << a+1 >> 3 behaves like ((1+2) << (a+1)) >> 3.
  • If the shift amount n exceeds the type width l, real hardware may effectively shift by n mod l; however, in C this still counts as undefined behavior.

A network mask is central to scalable networking because it abstracts many IP addresses into a single “network.” During Internet routing, routers mainly need to decide in which “direction” (that is, which network) the destination lies, while the exact host identity is handled only by edge devices close to the PC. 才关心。

⚠️ In simple terms, a router records the neighboring network prefixes, compares the destination against them through masks, and forwards the packet toward the approximate target network; the next router then repeats the same process.

All PCs (end hosts) sit behind edge devices such as home routers. Suppose router X defines subnet 192.168.50.0, takes address 192.168.50.1, and assigns the PC 192.168.50.123.0.123**. If the subnet contains fewer than 255 hosts, then the final 8 bits are enough to distinguish them, so a 24-bit mask (255.255.255.0) is sufficient. As you can see, ⚠️ the larger the mask, the smaller the subnet, because the masked-off portion is the host namespace within that subnet. Logical Operations

  • In C, every nonzero value counts as true. Values such as 0, \0, NULL, and nullptr are false, while values such as 42, "0", and "null" are true. For example, !!0x42 evaluates to true (0x1).
  • The logical operators are && (and), || (or), and ! (not). Do not confuse them with bitwise operations. They treat values as Booleans, not as bit patterns.
int a = 2, b = 10;
printf("%d, %d\n", a & b, a && b);

The code above prints `2, 1`, because `0010 & 1010 = 0010`, while `2 && 10 = 1` in Boolean logic.
  • ⚠️ A key feature of logical operators is short-circuit evaluation: expressions are evaluated from left to right, and once the overall truth value is already known, the rest is not evaluated. This is why && is commonly used as a safety guard in if / while conditions: ptr && *ptr == 10 will never dereference a null pointer, and i < n && !arr[i] prevents array-index overflow.
